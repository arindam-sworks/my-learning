Q01 Intermediate✓ Correct

Top 3 customers by total spend in the last 30 days

Table: orders(order_id, customer_id, amount, created_at)

Your answer

Your answer is correct. Logic, GROUP BY, ORDER BY, LIMIT — all fine. Only fix: typo cutomer_id → customer_id.

SQL — Final Answer
SELECT
    customer_id,
    SUM(amount) AS total_spend
FROM orders
WHERE created_at >= NOW() - INTERVAL 30 DAY  -- filter trailing 30 days; index-friendly
GROUP BY customer_id
ORDER BY total_spend DESC
LIMIT 3;
Explain

NOW() - INTERVAL 30 DAY preserves index usage. DATE_SUB(NOW(), INTERVAL 30 DAY) is equivalent. LIMIT always follows ORDER BY.

Quick recall

SQL execution order: WHERE → GROUP BY → HAVING → ORDER BY → LIMIT. Keep this sequence in mind — SQL processes clauses in this exact order.

Q02 Intermediate✓ Correct

Employees earning above their own department average

Table: employees(id, name, dept_id, salary)

Your answer

Correlated subquery is correctly written. Using e.dept_id to correlate with the outer query — perfect.

Bonus

Can also be solved with a window function — more efficient on large tables since it computes the department average in a single pass.

SQL — Correlated Subquery (your answer)
SELECT id, name, dept_id, salary
FROM employees e
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE dept_id = e.dept_id  -- correlated: uses outer row's dept_id
);
SQL — Window Function (optimised bonus)
SELECT id, name, dept_id, salary
FROM (
    SELECT *,
        AVG(salary) OVER (PARTITION BY dept_id) AS dept_avg
        -- computes avg for each department once across all rows
    FROM employees
) t
WHERE salary > dept_avg;
Correlated vs Window

Correlated subquery runs a separate inner query per row — effectively O(n²). Window function computes everything in one pass — much faster on large tables.

Quick recall

Correlated subquery = “inner query sees outer row”. Window function = “compute everything in one pass”. Both are valid; show the window function version as a bonus.

SELECT p.id, p.title
FROM products p
LEFT JOIN orders o ON o.product_id = p.id  -- bring all products; NULL where no match

SELECT p.id, p.title

Q04 Intermediate✓ Correct

Users with more than 3 sessions on a single calendar day

Table: sessions(user_id, login_at, logout_at)

Your answer

Perfect. DATE() to extract the date, GROUP BY, then HAVING — this is the correct pattern. Note: HAVING COUNT(*) > 3 correctly means four or more sessions.

SQL — Final Answer
SELECT
    user_id,
    DATE(login_at) AS login_date,
    COUNT(*) AS session_count
FROM sessions
GROUP BY user_id, DATE(login_at)  -- group by user + calendar day
HAVING COUNT(*) > 3;              -- filter after grouping (aggregate condition)
WHERE vs HAVING

WHERE runs before aggregation (filters individual rows). HAVING runs after aggregation (filters grouped results). COUNT(*) > 3 is an aggregate condition, so it belongs in HAVING.

Quick recall

If the condition involves an aggregate function (COUNT, SUM, AVG) → use HAVING. If it’s a plain column filter → use WHERE.

Q05 Intermediate✦ Missing

Pairs of employees in the same department with the same job title — self join

Not submitted

This query was not in your submission. Self-join is an important concept worth knowing for interviews.

SQL — Self Join
SELECT
    e1.id        AS emp1_id,
    e1.name      AS emp1_name,
    e2.id        AS emp2_id,
    e2.name      AS emp2_name,
    e1.dept_id,
    e1.job_title
FROM employees e1
JOIN employees e2
    ON  e1.dept_id   = e2.dept_id    -- same department
    AND e1.job_title = e2.job_title  -- same job title
    AND e1.id        < e2.id          -- prevent (A,B) and (B,A) duplication
ORDER BY e1.dept_id;
Why a self join?

Give the same table two different aliases and join it to itself. The e1.id < e2.id condition ensures each unordered pair appears exactly once — without it, every pair shows up twice.

Quick recall

Self join = call the same table by two names. Remember id1 < id2 to de-duplicate unordered pairs.

Q06 Intermediate✓ Correct + Bonus

Running total of transaction amounts per user

Cumulative sum of amount ordered by created_at, partitioned by user.

Your answer

Both approaches are correct — correlated subquery and window function. Showing the window function version is excellent.

SQL — Window Function (best answer)
SELECT
    user_id,
    created_at,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY created_at
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND CURRENT ROW  -- sum from first row up to current
    ) AS running_total
FROM transactions
ORDER BY user_id, created_at;
ROWS vs RANGE

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW uses physical row count. When there are ties (same timestamp), ROWS is safer. The default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

Quick recall

Running total = SUM() OVER (PARTITION BY user ORDER BY date). Showing the window function approach in interviews makes a strong impression.

Q08 Intermediate✓ Correct + Bonus

Second-highest salary — without LIMIT/OFFSET

Your answer

Both approaches are correct — nested MAX subquery and DENSE_RANK. The DENSE_RANK version handles tied salaries more cleanly.

Info

DENSE_RANK approach handles tied salaries correctly — all employees with the joint-second salary are returned.

SQL — Method 1: Nested MAX
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (
    SELECT MAX(salary) FROM employees  -- exclude the highest
);                                          -- then highest of the rest = 2nd
SQL — Method 2: DENSE_RANK (handles ties)
WITH ranked AS (
    SELECT
        emp_id, name, salary,
        DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
        -- 100,100,90 → DENSE_RANK: 1,1,2  |  RANK: 1,1,3  |  ROW_NUMBER: 1,2,3
    FROM employees
)
SELECT emp_id, name, salary
FROM ranked
WHERE rnk = 2;
RANK vs DENSE_RANK vs ROW_NUMBER

ROW_NUMBER: always unique (1,2,3,4) even with ties. RANK: gaps after ties (1,1,3). DENSE_RANK: no gaps (1,1,2) — most intuitive for “N-th highest” problems.

Quick recall

DENSE = no gap. Use DENSE_RANK for “2nd highest” questions — it handles ties correctly.

Q09 Intermediate⚠ Minor Fix

Orders where total line value exceeds ₹5,000

Tables: orders, order_items(order_id, product_id, qty, price)

Fix

The final INNER JOIN order_items ot should reference the CTE alias, not the original table name. Whatever name follows WITH ... AS is what you join on.

SQL — Fixed Answer
WITH order_totals AS (
    SELECT
        order_id,
        SUM(qty * price) AS total_price  -- total value per order
    FROM order_items
    GROUP BY order_id
)
SELECT o.order_id, ot.total_price
FROM orders o
INNER JOIN order_totals ot ON ot.order_id = o.order_id  -- join CTE alias, not original table
WHERE ot.total_price > 5000;
CTE naming

The name after WITH is your alias for that result set. You join on that name — not the original table. CTE = temporary named result set.

Quick recall

WITH name AS (...) — then use name exactly like a table in the outer query.

Q10 Intermediate✓ Correct

Pivot rows into columns — Math, Science, English marks per student

Table: scores(student_id, subject, marks)

Your answer

CASE WHEN + MAX() pivot technique — perfect. This is MySQL's standard approach since there is no native PIVOT keyword.

SQL — Conditional Aggregation Pivot
SELECT
    student_id,
    MAX(CASE WHEN subject = 'Math'    THEN marks END) AS Math,
    MAX(CASE WHEN subject = 'Science' THEN marks END) AS Science,
    MAX(CASE WHEN subject = 'English' THEN marks END) AS English
FROM scores
GROUP BY student_id;
Why MAX()?

For a student’s Math row, Science and English are NULL. MAX() ignores NULL and picks the real value. SUM also works but would add values if a student had multiple rows for the same subject. MAX is safer.

Quick recall

Pivot = CASE WHEN col = 'value' THEN metric END aggregated per group. MySQL has no PIVOT keyword — always use CASE.

Q20 Intermediate✦ New

GROUP_CONCAT — comma-separated list per group

Table: products(id, name, category_id), categories(id, name). Return each category with its product names as a single comma-separated string.

Why this matters

Almost every reporting or admin query at a product company needs a flat list inside a group. GROUP_CONCAT is the MySQL answer — it collapses many rows into one string. Interviewers test whether you reach for it instead of post-processing in application code.

SQL — Basic GROUP_CONCAT
SELECT
    c.name                                                           AS category,
    GROUP_CONCAT(p.name ORDER BY p.name SEPARATOR ', ')           AS product_list,
    COUNT(p.id)                                                     AS product_count
FROM categories c
LEFT JOIN products p ON p.category_id = c.id
GROUP BY c.id, c.name
ORDER BY c.name;
SQL — Bonus: DISTINCT + ORDER inside GROUP_CONCAT
-- Increase default 1024-byte limit for large concatenations
SET SESSION group_concat_max_len = 10000;

SELECT
    dept_id,
    GROUP_CONCAT(DISTINCT job_title
                  ORDER BY job_title
                  SEPARATOR ' | ')  AS unique_roles,
    GROUP_CONCAT(name
                  ORDER BY salary DESC
                  SEPARATOR ', ')    AS earners_high_to_low
FROM employees
GROUP BY dept_id;
Production gotcha

Default group_concat_max_len is 1024 bytes. Silent truncation happens when the concatenated string exceeds this — no error, just cut data. Always set the session variable for large payloads. MySQL 8.0.31+ supports LIMIT N inside GROUP_CONCAT.

Quick recall

GROUP_CONCAT(col ORDER BY col2 SEPARATOR ', '). Add DISTINCT to de-duplicate. Guard the max length with SET SESSION group_concat_max_len.

Q21 Intermediate✦ New

3-table JOIN — top customers by revenue with their city

Tables: orders(order_id, customer_id, amount), customers(id, name, city_id), cities(id, name). Return customers with total revenue > ₹10,000, their city, and order count.

Why this matters

Every previous question joins 2 tables at most. Senior interviews always include a 3- or 4-table JOIN with aggregation to check whether candidates know how to chain JOINs correctly, use meaningful aliases, and push filters to the right clause.

SQL — 3-Table JOIN with Aggregation
SELECT
    ci.name              AS city,
    cu.name              AS customer,
    COUNT(o.order_id)   AS order_count,
    SUM(o.amount)        AS total_revenue,
    ROUND(AVG(o.amount), 2) AS avg_order_value
FROM orders o
JOIN customers cu  ON cu.id = o.customer_id   -- orders → customers
JOIN cities    ci  ON ci.id = cu.city_id      -- customers → cities
GROUP BY ci.id, ci.name, cu.id, cu.name
HAVING SUM(o.amount) > 10000
ORDER BY total_revenue DESC
LIMIT 10;
SQL — Bonus: 4-table JOIN (add product category)
-- orders → order_items → products → categories
SELECT
    cat.name                 AS category,
    COUNT(DISTINCT o.order_id) AS orders,
    SUM(oi.qty * oi.price)    AS revenue
FROM orders o
JOIN order_items oi  ON oi.order_id   = o.order_id
JOIN products    p   ON p.id          = oi.product_id
JOIN categories  cat ON cat.id        = p.category_id
WHERE o.created_at >= '2024-01-01'
GROUP BY cat.id, cat.name
ORDER BY revenue DESC;
JOIN chain rule

Chain JOINs in data flow order: the driving table → first lookup → second lookup. Always include the non-driving table’s PK in GROUP BY (e.g. ci.id, cu.id) alongside the name — this prevents incorrect grouping if two cities or customers share a name.

Quick recall

3-table JOIN = chain the foreign keys in order. Use HAVING for the aggregate filter. Always GROUP BY the PK, not just the name.

Q11 Advanced✓ Correct

Top 2 products per category by sales — DENSE_RANK

Your answer

Excellent. GROUP BY + DENSE_RANK inside a CTE, then WHERE rnk <= 2 in the outer query — this is the industry-standard pattern for “top N per group” problems.

SQL — Final Answer
WITH ranked AS (
    SELECT
        category,
        product_id,
        SUM(sales) AS total_sales,
        DENSE_RANK() OVER (
            PARTITION BY category         -- separate rank per category
            ORDER BY SUM(sales) DESC    -- highest sales = rank 1
        ) AS rnk
    FROM products
    GROUP BY category, product_id
)
SELECT category, product_id, total_sales, rnk
FROM ranked
WHERE rnk <= 2;
Can window functions be used directly in WHERE?

No. Window functions cannot appear directly in WHERE. You must compute them first in a CTE or subquery, then filter in the outer query.

Quick recall

Pattern: window function in CTE → WHERE rnk <= N in outer query. This is the go-to for any “Top N per group” problem.

Q12 Advanced✓ Correct

Users who signed up and purchased within 24 hours — event funnel

Table: events(user_id, event_type, event_time)

Q13 Advanced✓ Correct

Recursive CTE — full path for every leaf category

Table: categories(id, name, parent_id)

Your answer

Recursive CTE structure is entirely correct — anchor member, recursive member, termination logic — all right.

SQL — Recursive CTE
WITH RECURSIVE category_path AS (

    -- ① Anchor: start from root nodes
    SELECT
        id, name, parent_id,
        name AS path              -- initial path = own name
    FROM categories
    WHERE parent_id IS NULL   -- no parent = root

    UNION ALL

    -- ② Recursive: find children and extend path
    SELECT
        c.id, c.name, c.parent_id,
        CONCAT(cte.path, ' > ', c.name) AS path
    FROM categories c
    JOIN category_path cte
        ON cte.id = c.parent_id
)

SELECT path FROM category_path
WHERE id NOT IN (
    SELECT DISTINCT parent_id
    FROM categories
    WHERE parent_id IS NOT NULL
);  -- nodes that are no-one's parent = leaf
How it works

1. Anchor = start from root (no parent). 2. Recursive step = join children to the growing CTE and extend the path string. 3. When there are no more children, recursion terminates. Leaf nodes are those whose id never appears as a parent_id.

Quick recall

Recursive CTE = Anchor UNION ALL Recursive. Use this for any tree or hierarchy traversal. Requires MySQL 8+.

Q14 Advanced✓ Correct

Most recent action per user from a large logs table

Table: logs(id, user_id, action, ts) — include index strategy.

Your answer

ROW_NUMBER approach is correct. Index suggestion including a covering index — excellent.

SQL — Final Answer
WITH ranked AS (
    SELECT
        user_id, action, ts,
        ROW_NUMBER() OVER (
            PARTITION BY user_id
            ORDER BY ts DESC    -- latest timestamp gets rank 1
        ) AS rn
    FROM logs
)
SELECT user_id, action, ts
FROM ranked
WHERE rn = 1;

-- Composite index for the window partition + order
CREATE INDEX idx_user_ts ON logs(user_id, ts DESC);

-- Covering index: includes 'action' so table lookup is avoided entirely
CREATE INDEX idx_covering ON logs(user_id, ts DESC, action);
Covering index

When all columns the query needs are present in the index leaf pages, MySQL skips the table row lookup entirely. This is a covering index — look for Extra: Using index in EXPLAIN output.

Quick recall

“Latest per group” = ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ts DESC), then filter WHERE rn = 1. DISTINCT ON (PostgreSQL) does not exist in MySQL.

Q15 Advanced✓ Correct

Month-over-month percentage revenue growth, handling zero-revenue months

Your answer

LAG() + NULLIF() + COALESCE combination — excellent. Edge cases for first month and zero-revenue months are handled correctly.

SQL — Final Answer
WITH monthly_revenue AS (
    SELECT
        DATE_FORMAT(created_at, '%Y-%m') AS month,
        SUM(amount) AS revenue
    FROM orders
    GROUP BY DATE_FORMAT(created_at, '%Y-%m')
),
with_prev AS (
    SELECT
        month, revenue,
        LAG(revenue, 1) OVER (ORDER BY month) AS prev_revenue
        -- LAG: bring previous row's value
    FROM monthly_revenue
)
SELECT
    month, revenue,
    COALESCE(prev_revenue, 0) AS prev_revenue,
    CASE
        WHEN prev_revenue IS NULL  THEN 'First Month'
        WHEN prev_revenue = 0     THEN 'No Prev Revenue'
        ELSE CONCAT(
            ROUND((revenue - prev_revenue)
                  / NULLIF(prev_revenue, 0) * 100, 2),
            '%'
        )  -- NULLIF: converts 0 to NULL, avoiding division-by-zero
    END AS growth_percent
FROM with_prev
ORDER BY month;
Why NULLIF?

NULLIF(x, 0) returns NULL if x equals 0. Dividing by NULL gives NULL (safe), while dividing by 0 raises an error. This is the standard division-by-zero guard.

Quick recall

LAG(col, 1) = previous row. LEAD(col, 1) = next row. LAG is essential for MoM and YoY growth calculations.

Q16 Advanced⚠ Syntax Fix

Count and total amount by payment status, plus grand total — ROLLUP

Table: payments(status, amount)

Fix

Double comma ,, after COALESCE — syntax error. Remove the extra comma and it is perfect.

SQL — Fixed Answer
SELECT
    COALESCE(status, 'Grand Total') AS status,
    -- ROLLUP adds a grand total row where status = NULL; label it
    COUNT(*) AS total_count,
    SUM(amount) AS total_amount

Q17 Advanced✓ Correct

Subscription lapse detection — gaps between periods

Your answer

Self-join for gap detection — correct. DATEDIFF > 1 to filter actual lapses — excellent.

Bonus

Can also be solved with a window function (LEAD), which is more elegant and avoids the potential pair-explosion of a self-join.

SQL — Method 1: Self Join (your answer)
SELECT
    s1.user_id,
    s1.end_date   AS gap_start,
    s2.start_date AS gap_end,
    DATEDIFF(s2.start_date, s1.end_date) AS gap_days
FROM subscriptions s1
JOIN subscriptions s2
    ON  s1.user_id    = s2.user_id
    AND s2.start_date > s1.end_date    -- next subscription starts after current ends
WHERE DATEDIFF(s2.start_date, s1.end_date) > 1  -- gap > 1 day = lapse
ORDER BY s1.user_id, s1.end_date;
SQL — Method 2: LEAD Window Function (elegant)
SELECT
    user_id,
    end_date   AS gap_start,
    LEAD(start_date) OVER (
        PARTITION BY user_id
        ORDER BY start_date
    ) AS gap_end
FROM subscriptions
HAVING DATEDIFF(gap_end, gap_start) > 1;
-- HAVING because window aliases are not available in WHERE

Quick recall

Gap = from current.end_date to next.start_date. LEAD(next_start) - current_end > 1 day = lapse exists.

Q18 Advanced✓ Correct

Delete duplicate rows — keep the lowest id per (email, phone)

Your answer

Both approaches are correct. You understood the extra subquery wrapper that MySQL requires — excellent.

SQL — Method 1: MIN(id) with derived table wrapper
DELETE FROM users
WHERE id NOT IN (
    SELECT keep_id FROM (   -- extra derived table wrap — MySQL quirk
        SELECT MIN(id) AS keep_id
        FROM users
        GROUP BY email, phone  -- keep one id per unique (email, phone)
    ) AS min_ids
);
SQL — Method 2: ROW_NUMBER (cleaner)
DELETE FROM users
WHERE id IN (
    SELECT id FROM (
        SELECT
            id,
            ROW_NUMBER() OVER (
                PARTITION BY email, phone
                ORDER BY id ASC       -- lowest id = rank 1 = keep
            ) AS rn
        FROM users
    ) ranked
    WHERE rn > 1  -- everything above rank 1 is a duplicate
);
MySQL quirk

In MySQL, you cannot reference the table you are deleting from in a subquery directly. Wrapping it in a derived table (an extra SELECT layer) makes MySQL treat it as a temporary result — this is the standard workaround.

Quick recall

In production, run a SELECT to verify the rows you will delete before issuing the DELETE. Use batch deletes (LIMIT 1000) on large tables to avoid long-held locks.

Q19 Advanced✓ Excellent

Collaborative filtering — top 5 unseen products from similar users

Your answer

4-step CTE pipeline — excellent! similar_users → recommended_products → ranked → top 5. This level of query will impress interviewers for senior roles.

SQL — Collaborative Filtering Pipeline
WITH similar_users AS (
    -- Step 1: find users who viewed the same products
    SELECT
        u1.user_id  AS target_user,
        u2.user_id  AS similar_user_id,
        COUNT(*) AS common_products
    FROM user_product_views u1
    JOIN user_product_views u2
        ON  u1.product_id = u2.product_id   -- same product viewed
        AND u1.user_id   != u2.user_id       -- different users
    GROUP BY u1.user_id, u2.user_id
),
recommended_products AS (
    -- Step 2: products seen by similar users but NOT by target
    SELECT
        su.target_user AS user_id,
        upv.product_id,
        SUM(upv.view_count * su.common_products) AS popularity_score
        -- more overlap with similar user = higher weight
    FROM similar_users su
    JOIN user_product_views upv
        ON upv.user_id = su.similar_user_id
    WHERE upv.product_id NOT IN (
        SELECT product_id
        FROM user_product_views v
        WHERE v.user_id = su.target_user  -- exclude what target already viewed
    )
    GROUP BY su.target_user, upv.product_id
),
ranked AS (
    -- Step 3: rank per user by score
    SELECT *,
        DENSE_RANK() OVER (
            PARTITION BY user_id
            ORDER BY popularity_score DESC
        ) AS rnk
    FROM recommended_products
)
-- Step 4: return top 5 per user
SELECT user_id, product_id, popularity_score, rnk
FROM ranked
WHERE rnk <= 5
ORDER BY user_id, rnk;
Context

Real production recommendation systems use Python or Spark, not raw SQL. But demonstrating this level of query in an interview shows deep mastery of complex JOINs, aggregation, and window functions — a strong signal for senior roles.

Quick recall

Complex problem — break it into CTE steps. Step 1: what do I need? Step 2: where from? Step 3: how to combine? Each CTE is one well-named step.

Q22 Advanced✦ New

Median salary — no MEDIAN() function in MySQL

Table: employees(id, dept_id, salary). Return the median salary overall, and per department.

Why this matters

MySQL has no built-in MEDIAN(). This is one of the most frequently asked senior-level questions precisely because it requires you to think mathematically and use ROW_NUMBER + COUNT correctly. Getting this wrong signals a gap in window function understanding.

SQL — Overall Median
WITH ranked AS (
    SELECT
        salary,
        ROW_NUMBER() OVER (ORDER BY salary)  AS rn,
        COUNT(*)     OVER ()                   AS total
    FROM employees
)
SELECT AVG(salary) AS median_salary
FROM ranked
WHERE rn IN (
    FLOOR((total + 1) / 2),  -- lower middle index
    CEIL ((total + 1) / 2)   -- upper middle index (same as lower when odd count)
);
-- For odd n=5: FLOOR=3, CEIL=3 → single middle row
-- For even n=6: FLOOR=3, CEIL=4 → AVG of rows 3 and 4 = true median
SQL — Median Per Department
WITH ranked AS (
    SELECT
        dept_id,
        salary,
        ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary) AS rn,
        COUNT(*)     OVER (PARTITION BY dept_id)                  AS total
    FROM employees
)
SELECT
    dept_id,
    AVG(salary) AS median_salary
FROM ranked
WHERE rn IN (
    FLOOR((total + 1) / 2),
    CEIL ((total + 1) / 2)
)
GROUP BY dept_id;  -- needed because even-count depts pick two rows
The maths

For n = 5 (odd): FLOOR(6/2) = 3, CEIL(6/2) = 3 → picks row 3, the true middle. For n = 6 (even): FLOOR(7/2) = 3, CEIL(7/2) = 4 → picks rows 3 & 4, AVG of those two = exact median.

Quick recall

Median = AVG of rows where rn IN (FLOOR((n+1)/2), CEIL((n+1)/2)). One CTE to add row numbers + total count; one outer query to average the middle rows.

Q23 Advanced✦ New

Running total and percentage of grand total — in one query

Table: transactions(user_id, created_at, amount). Each row should show: amount, cumulative total per user, this row as % of user’s total, and % of the overall grand total.

Why this matters

Q6 covered the running total in isolation. This question adds the second layer that trips senior candidates: combining a partitioned window (per user) with an unpartitioned window (grand total) in the same SELECT. Every analytics dashboard needs both.

SQL — Running Total + % of User Total + % of Grand Total
SELECT
    user_id,
    created_at,
    amount,

    -- Cumulative sum per user (resets at each new user)
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY     created_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    )                                                      AS running_total,

    -- This row's amount as % of this user's lifetime total
    ROUND(
        amount
        / NULLIF(SUM(amount) OVER (PARTITION BY user_id), 0)
        * 100, 2
    )                                                      AS pct_of_user_total,

    -- This row's amount as % of ALL transactions (no partition = entire table)
    ROUND(
        amount
        / NULLIF(SUM(amount) OVER (), 0)
        * 100, 2
    )                                                      AS pct_of_grand_total

FROM transactions
ORDER BY user_id, created_at;
Key difference between the two window specs

SUM() OVER (PARTITION BY user_id) = sum just for that user (denominator for % of user total). SUM() OVER () = empty window = sum across the entire result set = grand total. No ORDER BY in the frame = full partition sum, not cumulative.

Quick recall

Empty OVER () = whole table. OVER (PARTITION BY x) with no ORDER BY = full group sum. Always wrap divisors in NULLIF(…, 0).

Q24 Advanced✦ New

Islands & Gaps — find all consecutive login streaks per user

Table: logins(user_id, login_date DATE) — one or more rows per day. Find every streak of 3+ consecutive calendar days per user.

Why this matters

Q17 (subscription gap) uses DATEDIFF between two subscription periods. The “islands” version is harder: you have potentially hundreds of raw rows and must group consecutive ones without knowing their boundaries in advance. This is one of the most asked advanced SQL patterns at product companies (retention analytics, streak features, daily active user funnels).

The trick

For consecutive dates, date - ROW_NUMBER() days produces the same value for every date in a streak — that constant is the “island key”. Non-consecutive dates produce different constants and land in different groups.

SQL — Islands & Gaps
WITH deduped AS (
    -- one row per (user, day) even if multiple logins that day
    SELECT DISTINCT user_id, login_date
    FROM logins
),
numbered AS (
    SELECT
        user_id,
        login_date,
        -- Subtract sequential rank from the date.
        -- Consecutive dates produce the SAME constant → same island.
        DATE_SUB(login_date,
            INTERVAL (
                ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date) - 1
            ) DAY
        )                        AS island_key
    FROM deduped
)
SELECT
    user_id,
    MIN(login_date)  AS streak_start,
    MAX(login_date)  AS streak_end,
    COUNT(*)         AS streak_days
FROM numbered
GROUP BY user_id, island_key
HAVING COUNT(*) >= 3        -- only streaks of 3+ consecutive days
ORDER BY user_id, streak_start;
Worked example (user 7, dates shown)
login_date    rn    rn-1   island_key (date - (rn-1) days)
2024-01-01     1     0     2024-01-01   ← streak 1 starts
2024-01-02     2     1     2024-01-01   ←
2024-01-03     3     2     2024-01-01   ←
2024-01-05     4     3     2024-01-02   ← gap! new island
2024-01-06     5     4     2024-01-02   ← streak 2 starts

Quick recall

Island key = date - ROW_NUMBER() days. Group by (user_id, island_key), then MIN/MAX/COUNT to get streak boundaries. Add DISTINCT on the input if multiple events per day exist.

Section C — Write Patterns
Q25 Intermediate✦ New

UPSERT — INSERT … ON DUPLICATE KEY UPDATE

Every senior backend engineer writes upse rts daily. Interviewers test whether you know the three variants and their edge cases.

Why this matters

Every real-world system has counters, stock levels, or user settings that must be created if absent or updated if present. Writing this as two round-trips (SELECT then INSERT or UPDATE) introduces a race condition. The correct approach is a single atomic statement.

SQL — Variant 1: UPSERT with increment
-- page_views(page_id PK, view_count, last_viewed_at)
INSERT INTO page_views (page_id, view_count, last_viewed_at)
VALUES (42, 1, NOW())
ON DUPLICATE KEY UPDATE
    view_count     = view_count + VALUES(view_count),
    last_viewed_at = VALUES(last_viewed_at);
-- VALUES(col) = the value that was in the attempted INSERT row
-- On first visit: inserts (42, 1, now). On repeat: increments count.
SQL — Variant 2: Bulk UPSERT (overwrite)
-- Replace stock levels from a nightly feed
-- product_stock(product_id, warehouse_id, qty)  PK: (product_id, warehouse_id)
INSERT INTO product_stock (product_id, warehouse_id, qty, updated_at)
VALUES
    (101, 1, 50, NOW()),
    (102, 1, 30, NOW()),
    (101, 2, 20, NOW())
ON DUPLICATE KEY UPDATE
    qty        = VALUES(qty),    -- overwrite with new value
    updated_at = NOW();           -- always refresh timestamp
SQL — Variant 3: INSERT IGNORE (skip silently)
-- Only insert if not already subscribed — no update needed
INSERT IGNORE INTO user_subscriptions (user_id, plan_id, subscribed_at)
VALUES (99, 'pro', NOW());
-- Duplicate key → row silently skipped, no error, no update
StatementOn duplicate keyUse when
ON DUPLICATE KEY UPDATEUpdates specified columnsCounter increment, last-seen timestamp, stock sync
INSERT IGNORESilently skips the rowIdempotent seed data, de-duplication inserts
REPLACE INTODELETE old row + INSERT newRarely — auto_increment changes, cascades fire
Race condition warning

Always prefer ON DUPLICATE KEY UPDATE over application-level SELECT-then-INSERT/UPDATE. The database-level atomic statement eliminates the read-write race condition under concurrent load.

Quick recall

INSERT INTO … VALUES … ON DUPLICATE KEY UPDATE col = VALUES(col). VALUES(col) references the value from the attempted INSERT row. Use INSERT IGNORE when you only want to create, never overwrite.

Q26 Intermediate✦ New

Conditional UPDATE — CASE WHEN in SET, and UPDATE with JOIN

Apply different logic to different rows in a single UPDATE statement — no loops, no multiple round-trips.

Why this matters

Junior engineers reach for application-level loops or multiple UPDATE calls. Senior engineers know how to express conditional logic inside a single SQL statement. Interviewers look for this instinct when discussing batch jobs, pricing rules, or tier recalculations.

SQL — CASE WHEN inside SET
-- Apply tiered discounts to all active customers in one pass
UPDATE customers
SET
    discount_pct = CASE
        WHEN total_orders >= 50 THEN 20
        WHEN total_orders >= 20 THEN 15
        WHEN total_orders >= 10 THEN 10
        ELSE                       5
    END,
    updated_at   = NOW()
WHERE status = 'active';
SQL — UPDATE with JOIN (apply from another table)
-- Category-based price adjustment using a JOIN
UPDATE products p
JOIN categories c ON c.id = p.category_id
SET p.price = p.price * CASE
    WHEN c.name = 'Electronics' THEN 1.05   -- 5% increase
    WHEN c.name = 'Clothing'    THEN 0.95   -- 5% decrease
    ELSE                          1.00   -- no change
END
WHERE p.is_active = 1;
SQL — UPDATE from a subquery (recalculate derived column)
-- Sync orders.total_amount from order_items (recalculate derived column)
UPDATE orders o
JOIN (
    SELECT order_id, SUM(qty * price) AS calc_total
    FROM   order_items
    GROUP BY order_id
) totals ON totals.order_id = o.id
SET o.total_amount = totals.calc_total,
    o.updated_at   = NOW();
-- No WHERE needed — JOIN already limits to matching orders
Batch large updates

On multi-million-row tables, a single UPDATE locks all matching rows until it completes. Use LIMIT N in a loop (application-side) or a cursor (stored procedure) to process in batches of 5,000–10,000 rows and avoid long lock chains.

Quick recall

UPDATE t SET col = CASE WHEN … END WHERE … — one pass, no loop. For data from another table: UPDATE t1 JOIN t2 ON … SET t1.col = t2.col.
