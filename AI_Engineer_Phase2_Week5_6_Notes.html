LLM-as-Judge

The core insight: if you can't use humans to score thousands of outputs, use another LLM. A judge model evaluates your main model's outputs against explicit criteria. This unlocks scalable, automated, reproducible evaluation — the foundation of scientific prompt engineering.

Python — LLM judge implementation
import anthropic, json
client = anthropic.Anthropic()

def llm_judge(
    task_input:   str,
    model_output: str,
    ground_truth: str | None = None,
    criteria:     list[str]    = None,
) -> dict:
    """Score a model output 1–5 per criterion. Optionally compare to reference."""
    criteria = criteria or ["accuracy", "conciseness", "tone"]

    ref_block = f"""
<reference_answer>
{ground_truth}
</reference_answer>""" if ground_truth else ""

    prompt = f"""You are an expert evaluator scoring AI-generated outputs.

<task_input>{task_input}</task_input>
<model_output>{model_output}</model_output>
{ref_block}

Score the output 1–5 on each criterion. Be strict — use the full range.
1=Poor  2=Below avg  3=Average  4=Good  5=Excellent

Criteria: {criteria}

JSON ONLY:
{{
  "scores":    {{ {', '.join(f'"{c}": 1-5' for c in criteria)} }},
  "reasoning": {{ {', '.join(f'"{c}": "one sentence"' for c in criteria)} }},
  "overall":   1-5
}}"""

    resp = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=512,
        temperature=0.0,    # judges MUST be deterministic
        system="You are a strict, consistent evaluator. Never inflate scores.",
        messages=[{"role": "user", "content": prompt}]
    )
    return json.loads(resp.content[0].text)
Why temperature=0.0 for judges

A judge that gives different scores for identical inputs is useless. Always set temperature=0.0 for any evaluation call. Determinism is not optional — it's what makes comparisons meaningful. The same output scored twice should return the same score.

02

Judge Calibration & Bias Fixes

A naive LLM judge has two critical failure modes that silently corrupt your eval results. Understanding and fixing them is essential before trusting any judge-based eval.

Failure modeWhat happensFix
SycophancyJudge gives high scores to everything — especially long, fluent-sounding outputsExplicit rubric anchors + require critique on every output
Positional biasJudge favors whichever output appears first when comparing two optionsAlways test both orderings, average results
Verbosity biasLonger outputs score higher even if they are worsePenalize padding explicitly in rubric; score conciseness separately
Self-preferenceClaude-as-judge inflates Claude outputs; GPT-as-judge inflates GPT outputsUse a different model family as judge when possible
Python — Positional-bias-free comparison
def unbiased_ab_compare(inp: str, output_a: str, output_b: str) -> dict:
    """Judge both orderings, declare winner only if consistent."""
    r1 = _compare(inp, output_a, output_b)   # A first
    r2 = _compare(inp, output_b, output_a)   # B first (labels swap)

    w1 = r1["winner"]                              # "A" or "B"
    w2 = "A" if r2["winner"] == "B" else "B"       # re-normalize

    if w1 == w2:
        return {"winner": w1, "confidence": "high"}
    else:
        return {"winner": "tie", "confidence": "low"}

def _compare(inp: str, first: str, second: str) -> dict:
    prompt = f"""Compare two AI outputs. Pick the better one.

<input>{inp}</input>
<output_A>{first}</output_A>
<output_B>{second}</output_B>

JSON: {{"winner": "A" or "B", "reason": "one sentence"}}"""
    resp = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=128, temperature=0.0,
        messages=[{"role": "user", "content": prompt}]
    )
    return json.loads(resp.content[0].text)

# ── Meta-evaluation: measure judge quality vs human labels ────
def judge_quality(human_scores: list, judge_scores: list) -> dict:
    from scipy.stats import spearmanr
    import numpy as np
    r, p = spearmanr(human_scores, judge_scores)
    mae  = np.mean(np.abs(np.array(human_scores) - np.array(judge_scores)))
    return {
        "spearman_r": round(r, 3),
        "mae":         round(mae, 3),
        "quality":     "good" if r > 0.7 else "moderate" if r > 0.5 else "poor"
    }
# Target: Spearman r > 0.7 vs human labels. Below 0.5 → rethink rubric.

Designing Eval Sets

Your eval set is the most valuable asset in prompt engineering. A bad eval set gives you false confidence. A good one tells you the truth. Design it before writing any prompt variants — never after.

Ground Truth Generation Strategy

Python — Budget ground truth with strong model
def generate_ground_truth(inputs: list[str], task_description: str) -> list[str]:
    """
    Budget approach: use strongest model as pseudo-ground-truth.
    Validate on 10% with real human annotation.
    CRITICAL: run this BEFORE writing any test prompts.
    """
    results = []
    for inp in inputs:
        gt = client.messages.create(
            model="claude-opus-4-20250514",    # use strongest model available
            max_tokens=256,
            temperature=0.0,
            system="You are an expert. Produce the ideal, gold-standard output.",
            messages=[{"role": "user",
                       "content": f"Task: {task_description}\n\nInput:\n{inp}"}]
        ).content[0].text
        results.append(gt)
    return results

# The golden rule of eval set construction:
# Write ground truth → Lock it → THEN write and test prompt variants.
# If you peek at outputs before writing ground truth, you're measuring
# "how similar is my prompt to itself" — not actual quality.

# Inter-annotator agreement (when using human labelers):
# Get 3 humans to label independently, compute Cohen's kappa.
# Kappa < 0.6 → task definition is ambiguous → rewrite rubric first.

🧠 The eval set is your product's constitution

When you improve a prompt, the only way to know you've genuinely improved it — rather than accidentally overfit to your last test — is to measure against a fixed, locked eval set that predates your prompt changes. The eval set is not a test you pass; it's the objective specification of what "good" means for your task. Treat it with the same care as a database schema — changes need a migration, a reason, and a reviewer.

A/B Testing Prompt Variants Systematically

The core workflow: design variants that test specific hypotheses, run all on the same eval set, score with LLM judge, build a leaderboard. Change one thing at a time — not everything at once.

Python — A/B test framework
from dataclasses import dataclass
import time, numpy as np

@dataclass
class PromptVariant:
    name:            str
    system:          str
    user_template:   str   # use {input} as placeholder
    hypothesis:      str   # what you expect this to improve

@dataclass
class EvalResult:
    variant:    str
    case_id:    str
    output:     str
    scores:     dict
    latency_s:  float
    cost_usd:   float

def run_variant_on_case(v: PromptVariant, c: EvalCase) -> EvalResult:
    t0 = time.time()
    resp = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=256,
        system=v.system,
        messages=[{"role": "user",
                   "content": v.user_template.format(input=c.input)}]
    )
    output  = resp.content[0].text
    scores  = llm_judge(c.input, output, c.ground_truth,
                        ["accuracy", "conciseness", "actionability"])
    cost    = (resp.usage.input_tokens*3.0 + resp.usage.output_tokens*15.0) / 1e6
    return EvalResult(v.name, c.id, output, scores["scores"],
                      round(time.time()-t0,3), round(cost,6))

def build_leaderboard(results: list[EvalResult]) -> list[dict]:
    from itertools import groupby
    rows = []
    for name, grp in groupby(sorted(results, key=lambda r: r.variant),
                              key=lambda r: r.variant):
        g = list(grp)
        rows.append({
            "variant":      name,
            "accuracy":     np.mean([r.scores["accuracy"]     for r in g]).round(3),
            "conciseness":  np.mean([r.scores["conciseness"]  for r in g]).round(3),
            "actionability":np.mean([r.scores["actionability"]for r in g]).round(3),
            "overall":      np.mean([r.scores.get("overall",3)for r in g]).round(3),
            "avg_latency_s":np.mean([r.latency_s                for r in g]).round(3),
            "total_cost_usd":sum([r.cost_usd                   for r in g]),
        })
    return sorted(rows, key=lambda x: x["overall"], reverse=True)

The 5 Prompt Variants Pattern

Each variant should test one specific hypothesis. Don't change 5 things at once — you won't know what caused the improvement. Change one lever per variant.

Python — 5 hypothesis-driven variants (ticket summarization)
variants = [
    PromptVariant(
        name="v1_baseline",
        system="You are a helpful assistant.",
        user_template="Summarize this ticket in 1-2 sentences:\n\n{input}",
        hypothesis="Control — no engineering. Baseline to beat."
    ),
    PromptVariant(
        name="v2_role",
        system="You are a senior customer support manager who triages tickets quickly.",
        user_template="Summarize this ticket in 1-2 sentences:\n\n{input}",
        hypothesis="Does adding a specific role improve accuracy?"
    ),
    PromptVariant(
        name="v3_format",
        system="You are a senior support manager.",
        user_template="""Summarize the ticket using EXACTLY this format:
"[Issue]: [customer/context] reports [problem]. [Priority signal if present]."
Max 25 words. Ticket: {input}""",
        hypothesis="Does strict format improve conciseness score?"
    ),
    PromptVariant(
        name="v4_fewshot",
        system="You are a senior support manager.",
        user_template="""Summarize tickets in 1-2 sentences.

Ticket: "Order #556 is 2 weeks late. Very upset."
Summary: Customer reports 2-week delay on order #556. High urgency.

Ticket: "Love the product! Does it come in blue?"
Summary: Positive feedback with color inquiry. Low priority.

Ticket: {input}
Summary:""",
        hypothesis="Do examples help on ambiguous/edge-case tickets?"
    ),
    PromptVariant(
        name="v5_cot",
        system="You are a senior support manager.",
        user_template="""Analyze then summarize.

<analysis>
Type: [billing/shipping/technical/feedback/other]
Priority: [P1/P2/P3/P4]
Key facts: [bullet essentials]
</analysis>

<summary>[1-2 sentence summary for the queue]</summary>

Ticket: {input}""",
        hypothesis="Does structured CoT improve actionability score?"
    ),
]

🧠 One lever at a time

v2 changes only the system prompt role. v3 changes only the output format constraint. v4 adds only few-shot examples. v5 adds only CoT structure. If v5 wins, you know CoT helped — not role + format + examples + CoT combined. This is how you build a causal understanding of what your prompts are actually doing, rather than just a correlation between "I changed stuff" and "it got better."

Async Parallel LLM Calls

Running eval serially: 5 variants × 20 cases × 1 judge call = 100 API calls at ~1s each = 100s. With async: ~5–8s. For any eval with more than 10 calls, async is not optional.

Python · asyncio + AsyncAnthropic — Full async eval runner
import asyncio
from anthropic import AsyncAnthropic

async_client = AsyncAnthropic()

async def run_async(
    v:         PromptVariant,
    c:         EvalCase,
    sem:       asyncio.Semaphore,
) -> EvalResult:
    """Rate-limited async variant runner with judge call."""
    async with sem:
        t0 = time.time()

        # Main model call
        resp = await async_client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=256,
            system=v.system,
            messages=[{"role":"user", "content":v.user_template.format(input=c.input)}]
        )
        output = resp.content[0].text

        # Judge call (also async)
        judge_resp = await async_client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=256, temperature=0.0,
            messages=[{"role":"user", "content":build_judge_prompt(c, output)}]
        )
        scores = json.loads(judge_resp.content[0].text)
        cost   = (resp.usage.input_tokens*3+resp.usage.output_tokens*15)/1e6

        return EvalResult(v.name, c.id, output, scores["scores"],
                          round(time.time()-t0,3), round(cost,6))

async def run_full_eval(
    variants:       list[PromptVariant],
    eval_set:       list[EvalCase],
    max_concurrent: int = 10,
) -> list[EvalResult]:
    sem   = asyncio.Semaphore(max_concurrent)
    tasks = [run_async(v, c, sem) for v in variants for c in eval_set]

    print(f"Running {len(tasks)} tasks (max {max_concurrent} concurrent)...")
    results = await asyncio.gather(*tasks, return_exceptions=True)

    good = [r for r in results if isinstance(r, EvalResult)]
    bad  = [r for r in results if isinstance(r, Exception)]
    if bad: print(f"  {len(bad)} failed: {bad[0]}")
    return good

# Entry point
results = asyncio.run(run_full_eval(variants, eval_set))
Semaphore sizing

Set max_concurrent based on your API tier's rate limit. Claude's default is 50 requests/minute on the free tier, ~1000/min on paid. Start with max_concurrent=10 and increase if you're not hitting rate limits. Each "task" in the eval runs 2 API calls (main + judge), so 10 concurrent tasks = 20 in-flight requests.

Rate Limit Handling

Python — Exponential backoff on rate limits
from anthropic import RateLimitError, APIStatusError
import asyncio, random

async def call_with_backoff(
    prompt:      str,
    sem:         asyncio.Semaphore,
    max_retries: int   = 5,
    base_delay:  float = 1.0,
) -> str:
    """Async call with full jitter exponential backoff."""
    async with sem:
        for attempt in range(max_retries):
            try:
                resp = await async_client.messages.create(
                    model="claude-sonnet-4-20250514", max_tokens=256,
                    messages=[{"role": "user", "content": prompt}]
                )
                return resp.content[0].text

            except RateLimitError:
                # Full jitter: random(0, min(cap, base * 2^attempt))
                cap   = 60.0
                delay = random.uniform(0, min(cap, base_delay * (2 ** attempt)))
                print(f"Rate limit hit. Waiting {delay:.1f}s (attempt {attempt+1})")
                await asyncio.sleep(delay)

            except APIStatusError as e:
                if e.status_code == 529:   # Overloaded
                    await asyncio.sleep(30)
                else:
                    raise

        raise RuntimeError(f"Failed after {max_retries} retries")

# Token budget monitoring — track spend in real time
class BudgetGuard:
    def __init__(self, max_usd: float):
        self.max_usd = max_usd
        self.spent   = 0.0

    def record(self, input_tok: int, output_tok: int, model="sonnet"):
        cost = (input_tok*3 + output_tok*15) / 1e6
        self.spent += cost
        if self.spent > self.max_usd:
            raise RuntimeError(f"Budget exceeded: ${self.spent:.4f} > ${self.max_usd}")

budget = BudgetGuard(max_usd=2.00)   # hard cap at $2 per eval run

Prompt Chaining — When & How

A single prompt can't do everything well. Chaining breaks a complex task into steps where each step focuses on one thing, produces structured output, and passes it to the next. Use chains when a single prompt fails on complexity, when you need different models for different steps, or when each step's output needs validation before proceeding.

Python — 4-step blog post chain
from pydantic import BaseModel
from typing import List

class Requirements(BaseModel):
    goal: str; audience: str; tone: str
    key_points: List[str]; word_limit: int

class Research(BaseModel):
    key_facts: List[str]; examples: List[str]; data_points: List[str]

class Critique(BaseModel):
    weaknesses: List[str]; suggestions: List[str]; score: int

def chain_blog_post(raw_request: str) -> str:
    # Step 1: extract requirements (cheap model — just extraction)
    reqs = Requirements(**json.loads(
        client.messages.create(
            model="claude-haiku-4-20250307", max_tokens=512, temperature=0.0,
            messages=[{"role":"user",
                       "content":f"Extract: {Requirements.model_json_schema()}\n\n{raw_request}"}]
        ).content[0].text
    ))

    # Step 2: research (structured brainstorm)
    research = Research(**json.loads(
        client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=512,
            messages=[{"role":"user",
                       "content":f"Research material for: {reqs.model_dump_json()}\nJSON only: {Research.model_json_schema()}"}]
        ).content[0].text
    ))

    # Step 3: draft (uses both requirements and research)
    draft = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=2048,
        system=f"You are a {reqs.tone} writer for {reqs.audience}.",
        messages=[{"role":"user",
                   "content":f"Write ~{reqs.word_limit} words.\nReqs: {reqs.model_dump_json()}\nResearch: {research.model_dump_json()}"}]
    ).content[0].text

    # Step 4: critique (best model — quality gate)
    critique = Critique(**json.loads(
        client.messages.create(
            model="claude-opus-4-20250514", max_tokens=512,
            messages=[{"role":"user",
                       "content":f"Critique:\n{draft}\n\nReqs: {reqs.model_dump_json()}\nJSON: {Critique.model_json_schema()}"}]
        ).content[0].text
    ))

    # Conditional rewrite if score < 7
    if critique.score < 7:
        draft = client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=2048,
            messages=[{"role":"user",
                       "content":f"Rewrite incorporating critique:\n{draft}\n\nFix: {critique.weaknesses}\nApply: {critique.suggestions}"}]
        ).content[0].text

    return draft

Self-Critique & Refinement Loops

You can make the same model critique its own output and improve it — no separate judge needed. Less reliable than an external judge, but simpler. Know the failure modes.

Python — Self-critique loop implementation
def self_critique_loop(
    task:       str,
    input_text: str,
    criteria:   list[str] = None,
    max_rounds: int = 2,
    target:     float = 4.0,
) -> dict:
    criteria = criteria or ["accuracy", "clarity", "completeness"]

    # Round 0: initial output
    current = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=512,
        messages=[{"role":"user", "content":f"Task: {task}\n\nInput:\n{input_text}"}]
    ).content[0].text
    history = [{"round":0, "output":current}]

    for rnd in range(1, max_rounds+1):
        critique_resp = client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=512,
            messages=[{"role":"user", "content":f"""Critique this output.
Task: {task}
Output: {current}

Score each 1-5 and give SPECIFIC improvement suggestions.
Criteria: {criteria}
JSON: {{"scores":{{}}, "suggestions":{{}}, "overall": 1-5}}"""}]
        ).content[0].text

        crit = json.loads(critique_resp)
        history[-1]["critique"] = crit

        if crit["overall"] >= target:
            break   # good enough — stop

        current = client.messages.create(
            model="claude-sonnet-4-20250514", max_tokens=512,
            messages=[{"role":"user", "content":f"Rewrite based on critique:\nOutput: {current}\nCritique: {json.dumps(crit)}"}]
        ).content[0].text
        history.append({"round":rnd, "output":current})

    return {"final":current, "rounds":len(history), "history":history}

Statistical Significance in Evals

Without statistics, you can't tell if prompt A is genuinely better than B or if the 0.3 point difference on 20 cases is just noise. Statistical testing turns "seems better" into "is demonstrably better."

Null hypothesis: prompt A and prompt B have equal expected scores.
Paired t-test (same cases, both prompts): t = mean(A−B) / (std(A−B) / √n)
If p < 0.05 → reject null → the difference is statistically significant.
Python · scipy — Paired t-test + effect size
import numpy as np
from scipy import stats

def significance_test(
    scores_a: list[float],
    scores_b: list[float],
    alpha:    float = 0.05,
) -> dict:
    """
    Paired t-test — correct for eval sets where both prompts
    are tested on the SAME inputs (paired observations).
    NOT a two-sample test — those are for independent groups.
    """
    a, b  = np.array(scores_a), np.array(scores_b)
    diff  = a - b
    t, p  = stats.ttest_rel(a, b)

    mean_diff   = float(np.mean(diff))
    ci          = stats.t.interval(1-alpha, df=len(a)-1,
                                    loc=mean_diff, scale=stats.sem(diff))
    effect_size = mean_diff / float(np.std(diff, ddof=1))   # Cohen's d

    return {
        "mean_a":       round(float(np.mean(a)), 3),
        "mean_b":       round(float(np.mean(b)), 3),
        "mean_diff":    round(mean_diff, 3),
        "p_value":      round(float(p), 4),
        "significant":  bool(p < alpha),
        "ci_95":        (round(ci[0],3), round(ci[1],3)),
        "effect_size":  round(effect_size, 3),   # Cohen's d
        "verdict": (
            f"A significantly better (p={p:.3f})"  if p < alpha and mean_diff > 0 else
            f"B significantly better (p={p:.3f})"  if p < alpha and mean_diff < 0 else
            f"No significant difference (p={p:.3f})"
        )
    }

# Example run:
baseline_scores = [3.2, 2.8, 3.5, 3.0, 2.9, 3.8, 3.1, 3.4]
variant_scores  = [4.1, 3.9, 4.3, 3.8, 4.0, 4.5, 3.7, 4.2]
result = significance_test(baseline_scores, variant_scores)
print(result["verdict"])   # → "A significantly better (p=0.0003)"
Effect size guide (Cohen's d)

d = 0.2 → small effect (need 197 cases to detect reliably) · d = 0.5 → medium effect (32 cases) · d = 0.8 → large effect (13 cases). Most real prompt improvements fall in the 0.3–0.6 range — so plan for 30–50 cases minimum before drawing conclusions.

Sample Size Calculator

Python — Power analysis for eval planning
import numpy as np
from scipy.stats import norm

def required_sample_size(
    effect_size: float = 0.5,   # Cohen's d: 0.2=small 0.5=med 0.8=large
    alpha: float = 0.05,        # false positive rate (5% standard)
    power: float = 0.80,        # 80% chance of detecting a real effect
) -> int:
    z_a = norm.ppf(1 - alpha / 2)   # 1.96 for alpha=0.05
    z_b = norm.ppf(power)             # 0.84 for power=0.80
    return int(np.ceil(((z_a + z_b) / effect_size) ** 2))

# Planning table for AI eng eval work:
for d, desc in [(0.2,"small"),(0.3,"moderate"),(0.5,"medium"),(0.8,"large")]:
    n = required_sample_size(d)
    print(f"d={d} ({desc}): need {n} cases to reliably detect the difference")

# d=0.2 (small):    needs 197 cases
# d=0.3 (moderate): needs 90 cases
# d=0.5 (medium):   needs 34 cases
# d=0.8 (large):    needs 14 cases

# Practical guidance:
# 20 cases:  can only detect large improvements (clear wins only)
# 50 cases:  can detect medium improvements (typical in prod evals)
# 100+ cases: can detect small but meaningful improvements
