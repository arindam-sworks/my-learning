01

Why LangGraph Exists

In Weeks 9 and 10 you built agent loops manually — a for loop, a message list, some stopping conditions. That works for simple agents. But as agents grow more complex — multiple specialized steps, conditional branches, human approval gates, retry logic on failure — the raw loop becomes unmanageable. LangGraph solves this by representing your agent as an explicit directed graph.

Every node is a Python function. Every edge is a connection between nodes. The graph is the agent — you can visualize it, checkpoint it, resume it after failure, and inject human decisions at any point.

🧠 Graph = explicit agent logic

Your Week 9 agent loop was implicit — the logic lived inside a for loop with nested if statements. LangGraph makes that logic explicit as a graph you can visualize, serialize, checkpoint, and resume. When you say "go to the tools node if there's a tool call, go to human_review if confidence is low, otherwise end" — that IS the graph. The graph is the documentation of your agent's behavior.

02

LangGraph vs LangChain Chains

LangChain chains are linear pipelines — A → B → C. LangGraph is a graph — cycles, branches, conditional routing, interrupts. They solve different problems.

LangChain chains
Linear: A → B → C
No cycles — runs once and exits
Good for: RAG pipelines, simple transforms
No built-in state management
No checkpointing or resumability
Simpler to set up
LangGraph
Graph: cycles, branches, parallel paths
Loops — agent can revisit nodes
Good for: agents, multi-step reasoning
Typed state persisted across nodes
Checkpoints — pause, resume, branch
More setup, much more power
FeatureLangChain ChainLangGraph
Execution modelSequential, single passGraph traversal, cycles allowed
StatePassed forward as dictTyped schema, persisted, updatable
BranchingManual (RunnableBranch)First-class conditional edges
CheckpointingNot built-inBuilt-in (SQLite/Redis backends)
Human-in-loopNot supportedinterrupt_before / interrupt_after
Use whenFixed pipeline, no decisionsAgent loop, conditional logic, retries
When to use which

Use LangChain chains for your RAG pipeline (ingest → embed → retrieve → generate). Use LangGraph for your agent (decide → act → observe → decide again). The two compose well: your LangGraph agent can call a LangChain RAG chain as one of its nodes.

03

Nodes, Edges, and State

Three primitives. Master these and you can build any LangGraph agent.

Python — LangGraph minimal example
# pip install langgraph langchain-anthropic
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

# ── 1. STATE — typed dict that flows through all nodes ─────────
class AgentState(TypedDict):
    messages:  Annotated[list, operator.add]  # append-only (never overwrite)
    task:      str                            # the original task
    step:      int                            # how many steps taken
    done:      bool                           # termination signal
    result:    str                            # final output

# ── 2. NODES — pure functions: State → State update ────────────
def agent_node(state: AgentState) -> dict:
    """The LLM reasoning step. Returns a partial state update."""
    # ... call LLM, get response ...
    response = call_llm(state["messages"])

    if response.stop_reason == "end_turn":
        return {
            "messages": [response],
            "done":     True,
            "result":   response.content[0].text,
            "step":     state["step"] + 1,
        }
    return {
        "messages": [response],
        "step":     state["step"] + 1,
    }

def tools_node(state: AgentState) -> dict:
    """Execute all tool calls from the last message."""
    last   = state["messages"][-1]
    results = []
    for block in last.content:
        if block.type == "tool_use":
            result = dispatch_tool(block.name, block.input)
            results.append({"type":"tool_result",
                             "tool_use_id":block.id, "content":result})
    return {"messages": [{"role":"user","content":results}]}

# ── 3. CONDITIONAL EDGE — returns the next node name ───────────
def route_after_agent(state: AgentState) -> str:
    """Decide which node to go to next based on current state."""
    if state["done"]:
        return END
    if state["step"] >= 15:
        return END   # max steps reached
    last = state["messages"][-1]
    if hasattr(last, "stop_reason") and last.stop_reason == "tool_use":
        return "tools"
    return END

# ── 4. GRAPH ASSEMBLY ──────────────────────────────────────────
graph = StateGraph(AgentState)

# Add nodes
graph.add_node("agent", agent_node)
graph.add_node("tools", tools_node)

# Set entry point
graph.set_entry_point("agent")

# Add edges
graph.add_conditional_edges("agent", route_after_agent)  # conditional
graph.add_edge("tools", "agent")                        # always back to agent

# Compile
app = graph.compile()

# Run
result = app.invoke({
    "messages": [{"role":"user","content":"What is 2^32?"}],
    "task":     "What is 2^32?",
    "step":     0,
    "done":     False,
    "result":   "",
})
print(result["result"])
The key rules of nodes

Nodes must be pure functions: they take the current state and return a partial state update (a dict with only the keys that changed). LangGraph merges the update into the full state. Never mutate the input state directly — always return a new dict. The Annotated[list, operator.add] annotation tells LangGraph to append to the list rather than overwrite it — critical for message history.

State Schema Design

The state schema is the contract between all nodes. Design it carefully — a poorly designed state schema causes subtle bugs where nodes overwrite each other's work or produce incorrect behaviour when the graph branches.

Python — State schema patterns for real agents
from typing import TypedDict, Annotated, Optional, List
from langgraph.graph import StateGraph, END
import operator

# ── Pattern 1: APPEND-ONLY lists ──────────────────────────────
# Use Annotated[list, operator.add] for anything that should grow.
# If two parallel nodes both return {"messages": [x]},
# LangGraph merges them as [x1, x2] not x2 overwriting x1.

class CodeReviewState(TypedDict):
    # Immutable task inputs
    repo_url:       str
    target_files:   List[str]

    # Accumulated outputs (append-only)
    messages:       Annotated[list, operator.add]
    issues_found:   Annotated[List[dict], operator.add]  # each node appends issues
    file_contents:  Annotated[dict, lambda a, b: {**a, **b}]  # merge dicts

    # Mutable control flow state (nodes overwrite these)
    current_file:   Optional[str]
    step:           int
    phase:          str   # "clone" | "analyse" | "review" | "report"
    errors:         List[str]

    # Output
    final_report:   Optional[str]
    approved:       Optional[bool]   # set by human review node

# ── Pattern 2: REDUCER FUNCTIONS ──────────────────────────────
# Custom reducer: merge two dicts (for accumulating file contents)
def merge_dicts(existing: dict, update: dict) -> dict:
    return {**existing, **update}

# Custom reducer: keep highest-severity issues only
def keep_high_severity(existing: List[dict], new: List[dict]) -> List[dict]:
    combined = existing + new
    return [i for i in combined if i.get("severity") in ("high", "critical")]

# Usage in schema:
# issues: Annotated[List[dict], keep_high_severity]

# ── Pattern 3: NESTED STATE ────────────────────────────────────
# For complex state, nest TypedDicts
class ReviewMetadata(TypedDict):
    start_time:   str
    model_used:   str
    total_tokens: int

class FullReviewState(TypedDict):
    repo_url:   str
    issues:     Annotated[List[dict], operator.add]
    metadata:   ReviewMetadata   # nested TypedDict
    messages:   Annotated[list, operator.add]
State field typeReducer annotationWhen to use
Messages listAnnotated[list, operator.add]Any accumulating list — standard for messages
Dict mergingAnnotated[dict, lambda a,b: {**a,**b}]File contents, metadata accumulation
Simple overwriteNo annotationControl flags: step counter, phase, done bool
Custom reducerAnnotated[T, your_fn]De-duplication, filtering, max/min selection

Conditional Routing

Conditional edges are the decision points of your agent. A router function reads the current state and returns the name of the next node (or END). Multiple outputs from one router create a branching agent.

Python — Conditional routing patterns
from langgraph.graph import END

# ── Simple binary router ───────────────────────────────────────
def should_use_tools(state: AgentState) -> str:
    last = state["messages"][-1]
    if hasattr(last, "stop_reason") and last.stop_reason == "tool_use":
        return "tools"
    return END

# ── Multi-branch router ────────────────────────────────────────
def route_review_phase(state: CodeReviewState) -> str:
    phase = state["phase"]
    if phase == "clone":     return "read_files"
    if phase == "analyse":  return "llm_review"
    if phase == "review":   return "human_approval"
    if phase == "report":   return END
    return END  # fallback

# ── Safety router — always check for errors ───────────────────
def route_with_error_check(state: AgentState) -> str:
    if state.get("errors"):
        return "error_handler"
    if state["step"] >= 15:
        return "timeout_handler"
    last = state["messages"][-1]
    if hasattr(last, "stop_reason") and last.stop_reason == "tool_use":
        return "tools"
    return END

# ── Connecting edges to the graph ─────────────────────────────
graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tools_node)
graph.add_node("error_handler", error_handler_node)
graph.add_node("timeout_handler", timeout_handler_node)

graph.set_entry_point("agent")

# Conditional edge with explicit path_map
graph.add_conditional_edges(
    "agent",
    route_with_error_check,
    {
        "tools":           "tools",
        "error_handler":   "error_handler",
        "timeout_handler": "timeout_handler",
        END:               END,
    }
)
graph.add_edge("tools",           "agent")   # always back to agent
graph.add_edge("error_handler",   END)       # errors terminate
graph.add_edge("timeout_handler", END)
Use path_map for safety

Always pass the optional path_map dict to add_conditional_edges. This forces an explicit mapping of every possible return value to a node name, making routing errors fail loudly at compile time rather than silently routing to the wrong node at runtime. It also serves as documentation — the graph topology is visible in one place.

State Management in Long-Running Agents

LangGraph's checkpointer persists state to storage at every node. This enables resumability after failure, branching timelines, and inspection of every intermediate state — all essential for production agents.

Python — Checkpointing with SQLite and MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.checkpoint.memory import MemorySaver

# ── Development: in-memory checkpointer ───────────────────────
memory_checkpointer = MemorySaver()
app_dev = graph.compile(checkpointer=memory_checkpointer)

# ── Production: SQLite checkpointer ───────────────────────────
with SqliteSaver.from_conn_string("./agent_state.db") as checkpointer:
    app = graph.compile(checkpointer=checkpointer)

    # thread_id groups all turns of one conversation/task
    config = {"configurable": {"thread_id": "review-abc123"}}

    # First invocation
    result = app.invoke(initial_state, config=config)

    # Get the current state (snapshot after last node)
    snapshot = app.get_state(config)
    print(snapshot.values)          # full current state
    print(snapshot.next)            # which node runs next
    print(snapshot.config)          # thread_id + checkpoint_id

    # Resume after failure or interruption
    resumed = app.invoke(None, config=config)   # None = resume from checkpoint

    # Update state mid-run (inject human decision, fix a bug)
    app.update_state(config, {"approved": True, "phase": "report"})

    # List all checkpoints for this thread (full history)
    for checkpoint in app.get_state_history(config):
        print(checkpoint.config["configurable"]["checkpoint_id"],
              checkpoint.values.get("phase"))

# ── Streaming state updates ────────────────────────────────────
for event in app.stream(initial_state, config=config, stream_mode="updates"):
    # event is a dict: {node_name: state_update}
    for node_name, update in event.items():
        print(f"Node '{node_name}' updated: {list(update.keys())}")
thread_id is your agent session key

Every unique thread_id is a separate agent "conversation" with its own checkpoint history. Use a UUID per task run, or a stable ID (like a GitHub PR number) to allow resuming the same task. Multiple threads can run concurrently — checkpointers are thread-safe. In production, use PostgreSQL or Redis checkpointers instead of SQLite for horizontal scaling.

Human-in-the-Loop Checkpoints

Human-in-the-loop (HITL) is one of the most powerful features of LangGraph. You can pause the agent at any node — before executing a destructive action, before sending an email, before deploying code — and wait for human approval. The agent state is persisted while waiting. When the human responds, the graph resumes from exactly where it stopped.

Python — HITL with interrupt_before
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()

# Compile with interrupt_before: pause BEFORE these nodes run
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["human_approval"],    # pause here
    # interrupt_after=["dangerous_action"]  # or pause after a node
)

config = {"configurable": {"thread_id": "task-001"}}

# ── Step 1: Run until interrupt ────────────────────────────────
result = app.invoke(initial_state, config=config)
# Graph runs: clone_repo → analyse_files → llm_review
# PAUSES before human_approval node

snapshot = app.get_state(config)
print("Paused at:", snapshot.next)           # ('human_approval',)
print("Issues found:", snapshot.values["issues_found"])
# Agent is now waiting. Persist the config. Move on to other work.

# ── Step 2: Human reviews and decides ────────────────────────
# (minutes, hours, or days later...)
human_decision = {"approved": True, "human_comment": "LGTM, proceed"}
app.update_state(config, human_decision)

# ── Step 3: Resume from where we paused ──────────────────────
final = app.invoke(None, config=config)   # None = resume
# Graph continues: human_approval → generate_report → END
print(final["final_report"])

# ── FastAPI HITL endpoint pattern ─────────────────────────────
from fastapi import FastAPI
from pydantic import BaseModel

fastapi_app = FastAPI()

class ApprovalRequest(BaseModel):
    thread_id: str
    approved:  bool
    comment:   str = ""

@fastapi_app.post("/approve")
async def approve_action(req: ApprovalRequest):
    config = {"configurable": {"thread_id": req.thread_id}}
    app.update_state(config, {"approved": req.approved,
                               "human_comment": req.comment})
    # Resume in background
    import asyncio
    asyncio.create_task(asyncio.to_thread(app.invoke, None, config))
    return {"status": "resumed", "thread_id": req.thread_id}
PatternWhen to useLangGraph API
interrupt_beforeReview before a dangerous/irreversible actioncompile(interrupt_before=["node"])
interrupt_afterReview the output of a node before continuingcompile(interrupt_after=["node"])
update_stateInject human decision into paused stateapp.update_state(config, updates)
resumeContinue from checkpoint after human inputapp.invoke(None, config)
branchCreate alternative timeline from a checkpointapp.invoke(state, config_with_new_id)

Agent Tracing with LangSmith

LangSmith gives you a visual trace of every LLM call, every tool call, every token, and every latency in your agent run. For debugging complex agent failures — especially the kind that only happen one in twenty runs — it's indispensable.

Python — LangSmith setup and custom tracing
import os

# ── Setup: set env vars before any LangChain/LangGraph import ──
os.environ["LANGCHAIN_TRACING_V2"]  = "true"
os.environ["LANGCHAIN_API_KEY"]       = "ls__your_key"
os.environ["LANGCHAIN_PROJECT"]        = "code-reviewer-v1"
# All LangGraph/LangChain calls now automatically traced to LangSmith

# ── Run with trace metadata ────────────────────────────────────
from langsmith import traceable

@traceable(run_type="chain", name="code_review_run")
def run_code_review(repo_url: str) -> dict:
    config = {"configurable": {"thread_id": str(uuid.uuid4())}}
    result = app.invoke({
        "repo_url":      repo_url,
        "target_files":  [],
        "messages":      [],
        "issues_found":  [],
        "file_contents": {},
        "current_file":  None,
        "step":          0,
        "phase":         "clone",
        "errors":        [],
        "final_report":  None,
        "approved":      None,
    }, config=config)
    return result

# ── Custom spans for non-LangChain code ───────────────────────
from langsmith import trace

@traceable(run_type="tool", name="clone_repository")
def clone_repo_with_trace(repo_url: str) -> str:
    # Your tool implementation — automatically traced
    import subprocess
    result = subprocess.run(
        ["git", "clone", "--depth=1", repo_url, "./tmp_repo"],
        capture_output=True, text=True
    )
    if result.returncode != 0:
        raise RuntimeError(f"Git clone failed: {result.stderr}")
    return "./tmp_repo"

# ── What LangSmith shows you ──────────────────────────────────
# For each run:
# - Full graph traversal (which nodes ran, in what order)
# - Every LLM call: prompt, response, latency, token count
# - Every tool call: input, output, latency, errors
# - Total cost per run
# - Waterfall timeline (see what's slow)
# - Compare runs side-by-side
# - Create eval datasets from production traces
Three things LangSmith is essential for

1. Debugging flaky agents — agents that fail 1 in 10 runs are impossible to debug with print statements; LangSmith shows you exactly what the model said in that run. 2. Cost optimization — see which node consumes the most tokens and optimize it. 3. Eval dataset building — capture real production traces and convert them to eval cases for systematic testing.

Code Reviewer Agent — Architecture

Project 9 is a fully autonomous code reviewer: it clones a GitHub repo, reads files, analyses code quality, generates diff suggestions, and waits for human approval before finalizing the report.

Python — code_reviewer.py (full implementation)
import os, subprocess, ast, uuid
from pathlib import Path
from typing import TypedDict, Annotated, List, Optional
import operator
import anthropic
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver
from langsmith import traceable

os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_PROJECT"]     = "code-reviewer"

client = anthropic.Anthropic()

# ── STATE ─────────────────────────────────────────────────────
class ReviewState(TypedDict):
    repo_url:       str
    repo_path:      str
    files_to_review:List[str]
    file_contents:  Annotated[dict, lambda a, b: {**a, **b}]
    issues:         Annotated[List[dict], operator.add]
    messages:       Annotated[list, operator.add]
    current_file:   Optional[str]
    step:           int
    phase:          str
    errors:         List[str]
    approved:       Optional[bool]
    human_comment:  Optional[str]
    final_report:   Optional[str]

# ── NODES ─────────────────────────────────────────────────────
@traceable(run_type="tool", name="clone_repo")
def clone_repo_node(state: ReviewState) -> dict:
    repo_path = f"/tmp/review_{uuid.uuid4().hex[:8]}"
    result    = subprocess.run(
        ["git", "clone", "--depth=1", state["repo_url"], repo_path],
        capture_output=True, text=True
    )
    if result.returncode != 0:
        return {"errors": [f"Clone failed: {result.stderr}"], "phase": "error"}

    # Find .py files (limit to 10 for cost control)
    py_files = [str(p) for p in Path(repo_path).rglob("*.py")
                if ".git" not in str(p)][:10]
    return {"repo_path": repo_path, "files_to_review": py_files, "phase": "read"}

def read_files_node(state: ReviewState) -> dict:
    contents = {}
    for fpath in state["files_to_review"]:
        try:
            text = Path(fpath).read_text(encoding="utf-8", errors="ignore")
            contents[fpath] = text[:8000]   # truncate long files
        except: pass
    return {"file_contents": contents, "phase": "review"}

def llm_review_node(state: ReviewState) -> dict:
    file_summaries = "\n\n".join(
        f"### {path}\n{content[:2000]}"
        for path, content in state["file_contents"].items()
    )
    resp = client.messages.create(
        model="claude-sonnet-4-20250514", max_tokens=4096,
        system="""You are a senior code reviewer. Identify:
1. Security vulnerabilities (critical)
2. Performance issues (high)
3. Code quality issues (medium)
4. Style violations (low)
For each issue: file, line (estimate), severity, description, suggested fix.""",
        messages=[{"role":"user",
                   "content":f"Review this code:\n\n{file_summaries}\n\nReturn JSON list of issues."}]
    )
    import json, re
    raw = resp.content[0].text
    raw = re.sub(r'^```json\s*|\s*```$', '', raw, flags=re.M)
    try:
        issues = json.loads(raw)
    except:
        issues = [{"description": raw, "severity": "unknown"}]
    return {
        "issues":   issues,
        "messages": [resp],
        "phase":    "approve",
        "step":     state["step"] + 1,
    }

def human_approval_node(state: ReviewState) -> dict:
    """This node runs AFTER the human has approved via update_state."""
    if not state.get("approved"):
        return {"phase": "rejected"}
    return {"phase": "report"}

def generate_report_node(state: ReviewState) -> dict:
    issues_text = "\n".join(
        f"- [{i.get('severity','?').upper()}] {i.get('description','')}"
        for i in state["issues"]
    )
    report = f"""# Code Review: {state['repo_url']}

## Summary
Found {len(state['issues'])} issues in {len(state['files_to_review'])} files.

## Issues
{issues_text}

## Human Review Note
{state.get('human_comment', 'Approved.')}
"""
    return {"final_report": report, "phase": "done"}

def error_node(state: ReviewState) -> dict:
    return {"final_report": f"Failed: {state['errors']}"}

# ── ROUTING ───────────────────────────────────────────────────
def route_phase(state: ReviewState) -> str:
    phase = state["phase"]
    return {
        "read":     "read_files",
        "review":   "llm_review",
        "approve":  "human_approval",
        "report":   "generate_report",
        "rejected": END,
        "done":     END,
        "error":    "error_handler",
    }.get(phase, END)

# ── BUILD GRAPH ───────────────────────────────────────────────
graph = StateGraph(ReviewState)
for name, fn in [("clone_repo",clone_repo_node),("read_files",read_files_node),
                   ("llm_review",llm_review_node),("human_approval",human_approval_node),
                   ("generate_report",generate_report_node),("error_handler",error_node)]:
    graph.add_node(name, fn)
graph.set_entry_point("clone_repo")
graph.add_conditional_edges("clone_repo",     route_phase)
graph.add_conditional_edges("read_files",     route_phase)
graph.add_conditional_edges("llm_review",     route_phase)
graph.add_conditional_edges("human_approval", route_phase)
graph.add_conditional_edges("generate_report",route_phase)
graph.add_edge("error_handler", END)

app = graph.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["human_approval"],
)

🧠 Why you built agents from scratch before learning LangGraph

You spent two weeks building agent loops manually before touching LangGraph. Now that you're using it, you understand exactly what it's doing: the state dict is your messages list; the conditional edges are your if stop_reason == "tool_use" checks; the checkpointer is your trace logger. LangGraph adds persistence, visualization, and human-in-the-loop on top of the pattern you already know. Engineers who skip straight to frameworks spend months debugging things they don't understand.
