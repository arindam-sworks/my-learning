Vector Embeddings

An embedding is a dense numerical vector that encodes the meaning of text. Similar meanings produce vectors that point in similar directions. This is what makes semantic search possible — instead of matching keywords, you match meaning.

Python — Generating embeddings
import openai
from sentence_transformers import SentenceTransformer
import numpy as np

# ── Option 1: OpenAI embeddings (API, best quality) ────────────
oa = openai.OpenAI()

def embed_openai(texts: list[str], model="text-embedding-3-small") -> np.ndarray:
    resp = oa.embeddings.create(model=model, input=texts)
    return np.array([d.embedding for d in resp.data])

vecs = embed_openai(["The dog ran fast", "A quick canine sprinted"])
print(vecs.shape)   # (2, 1536)  — 1536 dims per text

# ── Option 2: Local embeddings (free, runs on CPU/GPU) ─────────
model = SentenceTransformer('all-MiniLM-L6-v2')   # 384 dims, very fast

texts = ["The dog ran fast", "A quick canine sprinted", "Python is a language"]
vecs  = model.encode(texts, normalize_embeddings=True)  # L2-normalize
print(vecs.shape)   # (3, 384)

🧠 Why embeddings work for retrieval

During training, an embedding model learns to place semantically similar text near each other in vector space. "The dog ran" and "a quick canine sprinted" never share keywords, but their meanings overlap — so their vectors end up close. This is the fundamental difference from keyword search: you're searching the meaning space, not the token space. The dot product (Week 1!) between a query vector and every document vector is the core retrieval operation.

Cosine Similarity & Nearest Neighbour Search

Cosine similarity measures the angle between two vectors, ignoring magnitude. It ranges from -1 (opposite) to 1 (identical direction). For normalized embeddings, cosine similarity equals the dot product — which is why dot product search is fast and effective.

cosine_similarity(A, B) = (A · B) / (‖A‖ × ‖B‖)
For L2-normalized vectors: cos_sim(A, B) = A · B
Python · NumPy — Similarity search from scratch
import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))

def top_k_search(
    query_vec:   np.ndarray,          # shape (dim,)
    doc_vecs:    np.ndarray,          # shape (n_docs, dim)
    doc_texts:   list[str],
    k:           int = 5,
) -> list[dict]:
    """Brute-force nearest neighbour — correct for small corpora."""
    # Normalize both for efficient dot product similarity
    q = query_vec / np.linalg.norm(query_vec)
    D = doc_vecs  / np.linalg.norm(doc_vecs, axis=1, keepdims=True)

    scores  = D @ q              # (n_docs,) — all similarities at once
    top_idx = np.argsort(scores)[::-1][:k]

    return [
        {"text": doc_texts[i], "score": round(float(scores[i]), 4), "idx": int(i)}
        for i in top_idx
    ]

# Scale: brute-force is O(n·d). Fine up to ~100k docs.
# Beyond that: use HNSW (ChromaDB default) or IVF-PQ (FAISS/Pinecone)
# HNSW: builds a hierarchical graph → O(log n) approximate search

# ── FAISS for large corpora (millions of docs) ─────────────────
import faiss

dim  = 384
index = faiss.IndexFlatIP(dim)     # Inner Product (dot product) index
# Normalize before adding (IP on unit vectors = cosine similarity)
faiss.normalize_L2(doc_vecs_np)
index.add(doc_vecs_np)             # add all docs
faiss.normalize_L2(query_np)
distances, indices = index.search(query_np.reshape(1,-1), k=10)  # top-10

# Trades a little accuracy for 100x+ speedup

Chunking Strategies

Chunking is how you split documents before embedding. It is the highest-leverage tunable parameter in a RAG system. Chunks too large → diluted embeddings, noisy context. Chunks too small → missing context, broken sentences. The right strategy depends on your document type.

Python — All three chunking strategies
from langchain.text_splitter import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter,
    MarkdownHeaderTextSplitter,
)
import re

# ── Strategy 1: FIXED-SIZE ─────────────────────────────────────
# Simple, fast, predictable. Splits at exact char count.
# Problem: splits mid-sentence, mid-paragraph — breaks context.
fixed = CharacterTextSplitter(
    separator="\n",
    chunk_size=500,    # ~125 words
    chunk_overlap=50,  # overlap prevents losing context at boundaries
)

# ── Strategy 2: RECURSIVE (default — use this for most cases) ──
# Tries to split on: paragraphs → sentences → words → chars
# Respects natural document structure. Produces coherent chunks.
recursive = RecursiveCharacterTextSplitter(
    chunk_size=1000,   # chars (~250 words) — sweet spot for most docs
    chunk_overlap=200, # 20% overlap — prevents losing context at edges
    separators=["\n\n", "\n", ". ", " ", ""],   # priority order
    length_function=len,
)

# ── Strategy 3: SEMANTIC (best quality, most expensive) ────────
# Groups sentences whose embeddings are similar into one chunk.
# Produces topically coherent chunks — better retrieval quality.
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

semantic = SemanticChunker(
    OpenAIEmbeddings(model="text-embedding-3-small"),
    breakpoint_threshold_type="percentile",  # split when similarity drops
    breakpoint_threshold_amount=85,           # 85th percentile of similarity
)

# ── Strategy 4: DOCUMENT-AWARE ─────────────────────────────────
# Split on meaningful structural boundaries — use for Markdown/HTML
md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[
        ("#",  "section"),
        ("##", "subsection"),
        ("###","subsubsection"),
    ],
    strip_headers=False,  # keep headers in chunk for context
)

# ── Add source metadata to every chunk ─────────────────────────
def chunk_document(text: str, source: str, page: int = 0) -> list[dict]:
    chunks = recursive.split_text(text)
    return [
        {
            "text":  chunk,
            "source": source,     # filename or URL
            "page":   page,
            "chunk_id": f"{source}::chunk_{i}",
            "char_count": len(chunk),
        }
        for i, chunk in enumerate(chunks)
    ]

The overlap rule

Always use chunk_overlap of 10–20% of chunk_size. Without overlap, a fact split across two chunks will be retrieved incompletely — the first chunk has the subject, the second has the predicate. Overlap ensures both versions contain the full sentence.

def ingest_pdf_dir(pdf_dir: str, collection: chromadb.Collection):
    import PyPDF2, os
    for fname in os.listdir(pdf_dir):
        if not fname.endswith(".pdf"): continue
        path = os.path.join(pdf_dir, fname)
        reader = PyPDF2.PdfReader(path)
        for page_num, page in enumerate(reader.pages):
            text   = page.extract_text()
            chunks = chunk_document(text, source=fname, page=page_num)
            # Add in batches of 100 (Chroma limit per call)

                batch = chunks[i:i+100]
                collection.add(
                    documents=[c["text"]     for c in batch],
                    ids=      [c["chunk_id"] for c in batch],
                    metadatas=[{"source":c["source"],"page":c["page"]}
                               for c in batch],
                )

Hybrid Search: BM25 + Vector

Vector search finds semantically similar text. BM25 finds exact keyword matches. Neither alone is best. Hybrid search combines both — it catches precise mentions (product names, IDs, technical terms) that vector search misses, while also catching paraphrases and synonyms that keyword search misses.

Python — BM25 + vector hybrid search with RRF
from rank_bm25 import BM25Okapi   # pip install rank-bm25
import numpy as np

class HybridSearcher:
    def __init__(self, chunks: list[dict], collection):
        self.chunks     = chunks
        self.collection = collection   # ChromaDB collection
        # Build BM25 index over tokenized chunk texts
        tokenized = [c["text"].lower().split() for c in chunks]
        self.bm25 = BM25Okapi(tokenized)

    def search(self, query: str, k: int = 10, alpha: float = 0.5) -> list[dict]:
        """
        alpha=0.0 → pure BM25
        alpha=1.0 → pure vector
        alpha=0.5 → equal blend (default)
        """
        # ── BM25 retrieval ────────────────────────────────────────
        bm25_scores = self.bm25.get_scores(query.lower().split())
        bm25_ranked = np.argsort(bm25_scores)[::-1][:k*2]  # get 2k candidates

        # ── Vector retrieval ──────────────────────────────────────
        vec_results = self.collection.query(
            query_texts=[query], n_results=k*2,
            include=["documents", "distances", "ids"]
        )
        # Map chunk_id → vector rank
        vec_ids   = vec_results["ids"][0]
        vec_rank  = {cid: rank for rank, cid in enumerate(vec_ids)}

        # ── Reciprocal Rank Fusion (RRF) ──────────────────────────
        K = 60   # RRF constant — reduces sensitivity to outlier ranks
        scores = {}
        for rank, idx in enumerate(bm25_ranked):
            cid = self.chunks[idx]["chunk_id"]
            scores[cid] = scores.get(cid, 0) + (1-alpha) / (K + rank)
        for cid, rank in vec_rank.items():
            scores[cid] = scores.get(cid, 0) + alpha / (K + rank)

        # Sort by fused score, return top-k
        top = sorted(scores.items(), key=lambda x: x[1], reverse=True)[:k]
        chunk_map = {c["chunk_id"]: c for c in self.chunks}
        return [
            {**chunk_map[cid], "hybrid_score": round(score, 6)}
            for cid, score in top if cid in chunk_map
        ]

# Why hybrid beats pure vector on technical docs:
# Query: "GPT-4 API rate limit 429"
# Vector: finds docs about API errors generally (semantic)
# BM25:   finds docs containing "429" and "GPT-4" exactly
# Hybrid: finds both — the exact API doc AND related context

Reranking with Cross-Encoders

Retrieval (BM25 + vector) is fast but approximate. A reranker takes the top-K retrieved chunks and scores each one more accurately — it reads the query AND the document together, producing a higher-quality relevance score. The two-stage pipeline is: retrieve 20–50 candidates cheaply, rerank to top-5 precisely.

Python — Cross-encoder + Cohere reranker
from sentence_transformers import CrossEncoder
import cohere

# ── Option 1: Local cross-encoder (free) ──────────────────────
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank_local(query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:
    pairs  = [(query, c["text"]) for c in candidates]
    scores = cross_encoder.predict(pairs)   # relevance score per pair
    ranked = sorted(zip(scores, candidates), key=lambda x: x[0], reverse=True)
    return [{**c, "rerank_score": float(s)} for s, c in ranked[:top_k]]

# ── Option 2: Cohere Rerank API (better, paid) ─────────────────
co = cohere.Client("your-cohere-api-key")

def rerank_cohere(query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:
    docs = [c["text"] for c in candidates]
    results = co.rerank(
        query=query,
        documents=docs,
        model="rerank-english-v3.0",
        top_n=top_k,
    )
    return [
        {**candidates[r.index], "rerank_score": r.relevance_score}
        for r in results.results
    ]

# ── Full two-stage retrieval pipeline ─────────────────────────
def retrieve_and_rerank(
    query:      str,
    searcher:   HybridSearcher,
    retrieval_k: int = 20,   # retrieve many cheaply
    rerank_k:    int = 5,    # rerank to few precisely
) -> list[dict]:
    # Stage 1: fast hybrid retrieval (20 candidates)
    candidates = searcher.search(query, k=retrieval_k)

    # Stage 2: precise reranking (5 best)
    reranked   = rerank_local(query, candidates, top_k=rerank_k)

    print(f"Retrieved {len(candidates)} → reranked to {len(reranked)}")
    return reranked

# Why reranking is worth the extra latency:
# Bi-encoder (embedding model): encodes query + doc SEPARATELY → fast but lossy
# Cross-encoder (reranker):     encodes query + doc TOGETHER  → slower but precise
# Cross-encoder sees full interaction between query and doc words
# This catches relevance signals that bi-encoders miss
Bi-encoder vs cross-encoder — the key difference

A bi-encoder embeds query and document independently and compares them with cosine similarity — fast enough to search millions of docs but misses subtle relevance. A cross-encoder reads query + document together as one input, so it captures every interaction between query words and doc words — much more accurate but too slow to run on the whole corpus. The two-stage pattern is the industry standard: retrieve wide, rerank narrow.

RAG Evaluation with RAGAS

RAGAS (RAG Assessment) provides a suite of metrics that measure different aspects of RAG quality — without requiring human labels for most of them. You need to hit good scores on all three core metrics before shipping a RAG system.

Faithfulness
Are the claims in the answer supported by the retrieved context? Measures hallucination.
Target: > 0.85
Answer Relevance
Does the generated answer actually address the question that was asked?
Target: > 0.80
Context Recall
Did retrieval find all the information needed to answer correctly? Measures retrieval quality.
Target: > 0.80

Context Window Management in RAG

Context management is what separates a demo RAG from a production RAG. You need to fit retrieved chunks, conversation history, system prompt, and output budget inside the model's context window without exceeding it or wasting tokens on irrelevant content.

Python — Context budget manager
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

def count_tokens(text: str) -> int:
    return len(enc.encode(text))

class ContextBuilder:
    def __init__(self, model_context: int = 200_000):
        self.model_context = model_context
        self.system_budget = 500    # tokens for system prompt
        self.output_budget = 1500   # tokens reserved for answer
        self.history_budget= 2000   # tokens for conversation history
        self.available     = (model_context
                              - self.system_budget
                              - self.output_budget
                              - self.history_budget)

    def build_context(
        self,
        chunks:  list[dict],   # already reranked, best first
        query:   str,
        history: list[dict] = None,
    ) -> tuple[str, list[dict]]:
        """
        Pack as many chunks as fit in the available token budget.
        Best chunks (index 0) are always included.
        Returns context string + the chunks actually used.
        """
        history   = history or []
        used_tok  = count_tokens(query)
        used_chunks = []
        context_parts = []

        for chunk in chunks:
            chunk_tok = count_tokens(chunk["text"])
            if used_tok + chunk_tok > self.available:
                break   # no more room
            context_parts.append(
                f"[Source: {chunk['source']}, page {chunk.get('page', '?')}]\n"
                + chunk["text"]
            )
            used_chunks.append(chunk)
            used_tok += chunk_tok

        context_str = "\n\n---\n\n".join(context_parts)
        print(f"Using {len(used_chunks)}/{len(chunks)} chunks ({used_tok} tokens)")
        return context_str, used_chunks

    def truncate_history(self, history: list[dict]) -> list[dict]:
        """Keep most recent history that fits in history_budget."""
        total, kept = 0, []
        for msg in reversed(history):
            tok = count_tokens(msg["content"])
            if total + tok > self.history_budget: break
            kept.insert(0, msg); total += tok
        return kept
The "lost in the middle" problem — revisited

Research shows LLMs retrieve information most reliably from the start and end of context. For RAG: always put your highest-ranked (most relevant) chunks at the beginning of the context block. If you're adding conversation history, put it at the very end, after the retrieved documents. Never bury your best evidence in the middle of 50 chunks.

Full RAG Pipeline — End to End

Putting it all together: ingestion, retrieval, generation, citation, and a FastAPI endpoint.

Python — Complete RAG system
import anthropic
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List

client = anthropic.Anthropic()

class Citation(BaseModel):
    source: str; page: int; excerpt: str

class RAGResponse(BaseModel):
    answer: str; citations: List[Citation]; chunks_used: int

RAG_SYSTEM = """You are a precise document Q&A assistant.
Answer questions using ONLY the provided context.
After each factual claim, cite the source: [source.pdf, p.N]
If the answer is not in the context, say exactly:
"This information is not available in the provided documents."
Never use outside knowledge."""

def generate_answer(
    question:    str,
    chunks:      list[dict],
    history:     list[dict] = None,
) -> RAGResponse:
    builder  = ContextBuilder()
    ctx, used = builder.build_context(chunks, question)
    hist     = builder.truncate_history(history or [])

    messages = hist + [{
        "role": "user",
        "content": f"""<context>
{ctx}
</context>

Question: {question}"""
    }]

    resp = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1500,
        temperature=0.1,    # low temp for factual RAG
        system=RAG_SYSTEM,
        messages=messages,
    )
    answer = resp.content[0].text

    # Extract citations from used chunks
    citations = [
        Citation(
            source=c["source"],
            page=c.get("page", 0),
            excerpt=c["text"][:150] + "..."
        )
        for c in used
    ]
    return RAGResponse(answer=answer, citations=citations, chunks_used=len(used))

# ── FastAPI endpoint ───────────────────────────────────────────
app = FastAPI(title="RAG API")

    question: str
    history:  List[dict] = []

        yield f"data: {json.dumps({'type':'citations','data':citations_payload})}\n\n"
        # Stream the answer
        with client.messages.stream(
            model="claude-sonnet-4-20250514", max_tokens=1500,
            temperature=0.1, system=RAG_SYSTEM,
            messages=[{"role":"user","content":f"<context>{ctx}</context>\n\nQuestion: {req.question}"}]
        ) as stream:
            for chunk in stream.text_stream:
                yield f"data: {json.dumps({'type':'token','data':chunk})}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")

🧠 What makes this a "production" RAG vs a demo

A demo RAG calls embeddings on a handful of docs and queries ChromaDB. A production RAG has hybrid search (BM25 + vector catches what pure semantic search misses), a reranker (selects the best 5 from 20 candidates), context window management (never overflows the LLM context silently), citation tracking (every claim is traceable to a source chunk), and RAGAS scores (you know numerically how good the system is). Project 6 forces you to implement all of these. When an interviewer asks "have you built a RAG system?" — this is the answer that earns respect.

Retrieval is nearest-neighbour search over those vectors. Reranking is a cross-encoder — a transformer (Week 3) scoring query+doc pairs. RAGAS evaluation applies LLM-as-judge (Week 5–6). Context management is token counting (Week 3). The entire RAG stack is built from foundations you already understand — it's all connected.
