Chapter 1 — Internals & Data Types
T1 Core

Why Redis is extremely fast — single-threaded architecture, in-memory storage, I/O model

Redis = REmote DIctionary Server. An in-memory data structure store used as cache, database, message broker, and streaming engine. A single Redis instance on modern hardware handles 100,000+ ops/sec with sub-millisecond latency.

FactorDetailImpact
In-memory storageAll data in RAM (100ns access vs disk milliseconds)Foundation of all speed gains
Single-threaded event loopOne thread executes commands — no mutex, no context switchingEliminates lock contention; commands are atomic
I/O multiplexingepoll/kqueue handles thousands of connections on one threadHigh concurrency without thread overhead
Efficient data structuresSkip lists, hash tables, intsets, ziplist/listpackO(1) or O(log n) for most operations
RESP protocolSimple text protocol, minimal parsing overheadFast serialization on both ends
text — Redis architecture
Redis Process (single-threaded event loop)
  ├── epoll/kqueue  — multiplexes thousands of client connections
  ├── Command parser — RESP protocol
  ├── Data structures — all in RAM
  └── Background threads — disk I/O, AOF rewrite, lazy free

Single-threaded = no locks = no context switch overhead
I/O multiplexing = one thread handles 10,000+ connections

Redis 6+: I/O threading added
  Main thread: command execution (still single-threaded)
  I/O threads: reading/writing network data in parallel
Say this in the interview

"Redis is fast because everything lives in RAM and the single-threaded architecture eliminates lock contention. The single-threaded model means commands are atomic by default — no partial writes, no race conditions on individual commands. In Redis 6+, I/O threading improves network throughput without changing the single-threaded command execution model."

Quick recall

RAM access = 100ns. Single thread = no locks + atomic commands. epoll multiplexing = 10k+ connections on one thread. Redis 6+ added I/O threads for network, not for command execution. Typical latency: <1ms. Throughput: 100k+ ops/sec.

T3

String commands in depth — SET options, atomic operations, counters

Strings are the most fundamental Redis type, but they carry rich semantics beyond simple key-value storage: conditional sets, atomic increments, TTL in one command, and binary bitmaps.

redis-cli — String commands
# Basic SET with options
SET key value                        # simple set
SET key value EX 3600               # expire in 3600 seconds
SET key value PX 5000               # expire in 5000 milliseconds
SET key value NX                     # set only if NOT exists (atomic!)
SET key value XX                     # set only if EXISTS
SET key value EX 3600 NX            # set with TTL only if not exists — distributed lock pattern
SET key value GET                    # return old value while setting new one

# Atomic counters
INCR        counter                  # atomic +1
INCRBY      counter 5               # atomic +5
DECR        counter
INCRBYFLOAT price 0.99

# Bulk operations
MSET   k1 v1 k2 v2 k3 v3            # set multiple (atomic)
MGET   k1 k2 k3                      # get multiple in one round-trip
MSETNX k1 v1 k2 v2                  # set multiple only if NONE exist

# String as bit array (Bitmap)
SETBIT   user:flags:123 0 1         # bit 0 = email verified
GETBIT   user:flags:123 0
BITCOUNT user:logins:2024            # count active days
Key pattern: SET NX

SET key value EX 3600 NX is the atomic pattern for distributed locks and idempotent operations — sets a key with TTL only if it doesn't exist, in a single command. No race condition between check and set.

Performance note

Always prefer MGET/MSET over individual GET/SET in loops. Each Redis round-trip is ~0.5ms on localhost. 100 individual GETs = 50ms. One MGET = 0.5ms.

Quick recall

SET NX = atomic "set if not exists". INCR = thread-safe counter (no race condition). MGET/MSET = batch for performance. EX/PX = TTL in seconds/milliseconds. GET flag returns old value. Bitmap via SETBIT/BITCOUNT for bit-level flags.

Lists, Sets, and Sorted Sets — commands and real-world use cases

redis-cli — List, Set, Sorted Set commands
── LIST (ordered, duplicates allowed) ──

RPUSH  queue job3               # push to tail

BRPOP  queue 0                 # BLOCKING pop — waits until item available (task queue!)

── SET (unordered, unique members) ──
SADD      tags:post:1 redis nosql cache
SMEMBERS  tags:post:1              # get all members
SISMEMBER tags:post:1 redis        # O(1) membership check
SINTER    tags:post:1 tags:post:2  # intersection
SUNION    tags:post:1 tags:post:2  # union
SDIFF     tags:post:1 tags:post:2  # difference

── SORTED SET (unique, ordered by float score) ──
ZADD         leaderboard 1500 player1 2200 player2 980 player3
ZRANK        leaderboard player1    # rank 0-indexed (low to high)
ZREVRANK     leaderboard player2    # rank high to low (position 0 = top)
ZSCORE       leaderboard player1
ZINCRBY      leaderboard 100 player1

BRPOP is the hidden gem

BRPOP is the secret weapon for simple task queues — it blocks until an item is available, giving zero-poll-latency. No need for polling loops or sleep() calls in your consumer. LPUSH + BRPOP = a complete task queue in two commands.

Quick recall

List = ordered with duplicates, BRPOP for blocking consumer. Set = unordered unique, SINTER/SUNION/SDIFF for set math. Sorted Set = unique with float score, ZADD/ZRANK/ZRANGE for ranked data. Sorted Set with score=timestamp = sliding window rate limiter. Sorted Set with score=priority = priority queue.

T6 Core

TTL, key expiry & eviction policies — what happens when Redis runs out of memory

Redis expires keys using two complementary mechanisms: lazy expiry (check on access) and active expiry (background sampling of 20 random keys per cycle). The combination means expired keys are cleaned up efficiently without blocking the event loop.

redis-cli — TTL commands
# Set expiry
EXPIRE   session:abc 3600       # expire in 3600 seconds
EXPIREAT session:abc 1735689600 # expire at Unix timestamp
PEXPIRE  session:abc 5000       # expire in milliseconds

# Inspect TTL
TTL  session:abc                 # seconds remaining (-1 = no expiry, -2 = key gone)
PTTL session:abc                 # milliseconds remaining

# Remove expiry (make permanent)
PERSIST session:abc

# Expiry mechanisms:
# 1. Lazy: check on access — expired? delete + return nil
# 2. Active: background cycle tests 20 random keys/second
#    If >25% expired → run cycle again (adaptive frequency)
Policy (maxmemory-policy)EvictsBest for
noevictionNothing — returns error on write when fullPrimary DB (can't lose data)
allkeys-lruLeast recently used from all keysGeneral cache (most common)
allkeys-lfuLeast frequently used from all keysHot/cold data with clear usage patterns
volatile-lruLRU among keys WITH expiry setMixed cache + persistent data in same instance
volatile-ttlShortest TTL first among expiring keysEvict soon-to-expire keys first
allkeys-randomRandom key from all keysUniform access patterns
Production guidance

For a pure cache: allkeys-lru. For a session store where active sessions must not be evicted: volatile-lru (only expired sessions get evicted). Always set maxmemory to 70–80% of available RAM to leave headroom for fork() during RDB snapshots.

Quick recall

TTL -1 = no expiry. TTL -2 = key doesn't exist. Lazy + active expiry = efficient cleanup without blocking. allkeys-lru = most common cache policy. volatile-lru = safe for mixed cache/persistent. Set maxmemory with buffer — hitting 100% causes OOM evictions or errors depending on policy.

Persistence — RDB snapshots vs AOF, trade-offs and production config

FeatureRDB (snapshot)AOF (append-only file)
What it storesPoint-in-time binary snapshot of entire datasetEvery write command, append-only log
Data loss on crashUp to last snapshot (minutes)Max 1 second (with fsync=everysec)
Startup speedFast — load binary snapshotSlower — replay all commands
File sizeCompact binary formatGrows large; AOF rewrite shrinks it
Performance impactfork() on snapshot — brief pause on large datasetsConstant small overhead per write
Best forBackups, fast restart, cache snapshotsSessions, queues — any data you can't lose
config — redis.conf persistence settings
# RDB — save if N keys changed in M seconds
save 900 1       # after 900s if ≥1 key changed
save 300 10      # after 300s if ≥10 keys changed
save 60  10000  # after 60s if ≥10000 keys changed

# AOF
appendonly yes
appendfsync everysec    # fsync every second — recommended (max 1s data loss)
# appendfsync always  → fsync every write (max durability, slow)
# appendfsync no      → OS decides (fastest, most data loss risk)

# AOF rewrite — compress the log when it grows too large
auto-aof-rewrite-percentage 100  # rewrite when AOF doubles in size
auto-aof-rewrite-min-size 64mb

# Best of both worlds — recommended for production:
appendonly yes
appendfsync everysec
save 900 1    # AOF for durability + RDB for fast restart
Production recommendation

Enable both RDB and AOF together. AOF with fsync=everysec gives max 1 second data loss. RDB provides fast restart snapshots. Pure cache? Disable persistence entirely for maximum throughput. Monitor rdb_last_bgsave_status and watch for fork() latency spikes on large datasets.

Quick recall

RDB = snapshot, fast restart, up to minutes of data loss. AOF = command log, slow replay, max 1s loss with everysec. Run both in production. Cache-only? Disable persistence. fork() for RDB runs in child process — doesn't block commands but can spike RSS memory usage.

T8 Advanced

Transactions — MULTI/EXEC, WATCH optimistic locking, and Lua scripts

MULTI/EXEC queues commands and executes them as an atomic block. Unlike SQL transactions, Redis transactions do not roll back on command failure — if SET succeeds but INCR fails due to wrong type, the other commands still execute.

redis-cli — MULTI/EXEC, WATCH, Lua
── MULTI/EXEC: queue and execute atomically ──
MULTI                           # start transaction
INCR account:balance
DECRBY account:balance 50      # queued, not executed yet
EXEC                            # execute all as atomic block
# ⚠ Not rollback-able! Individual command errors don't abort others

── WATCH: optimistic locking (CAS) ──
WATCH account:balance           # monitor this key for changes
GET   account:balance           # read current value
MULTI
DECRBY account:balance 100
EXEC                            # returns nil if balance changed since WATCH!
# Retry the whole block if EXEC returns nil

── Lua script: preferred for complex atomicity ──
# Atomic rate limiting in Lua (entire script runs without interruption)
EVAL "
local current = redis.call('INCR', KEYS[1])
if current == 1 then
  redis.call('EXPIRE', KEYS[1], ARGV[1])
end
if current > tonumber(ARGV[2]) then
  return 0
end
return 1
" 1 ratelimit:user:123 60 100
# Args: 1 key, ttl=60s, max=100 requests → returns 0 (blocked) or 1 (allowed)

# Register script for reuse (avoid re-sending script body each call)
SCRIPT LOAD "..."               # returns SHA1
EVALSHA <sha1> 1 key arg1       # call by SHA — much faster
When to use Lua vs MULTI/EXEC

Use Lua when you need true read-modify-write atomicity (check-then-set, rate limiting, distributed locks). The entire Lua script executes atomically in the single-threaded Redis process — no other commands can interleave. WATCH+MULTI is fine for simple CAS operations with low contention, but Lua is superior for high-contention scenarios.

Quick recall

MULTI/EXEC = atomic execution but no rollback. WATCH = CAS (compare-and-swap): EXEC returns nil if watched key changed → retry loop. Lua = true atomic read-modify-write, no other commands can interleave. EVALSHA + SCRIPT LOAD = efficient repeated Lua calls. Use Lua for rate limiting, locks, conditional updates.

Pub/Sub & Streams — fire-and-forget vs persistent messaging

FeaturePub/SubStreams
PersistenceFire-and-forgetStored in Redis
Offline consumersMiss messages if offlineRead from any point in history
Consumer groupsAll subscribers get all messagesEach group gets unique messages (parallel workers)
AcknowledgementNo delivery guaranteeXACK confirms processing
ReplayNot possibleReplay from any ID
Best forLive notifications, chat, presence updatesEvent queues, reliable messaging, audit logs
redis-cli — Pub/Sub and Streams
── PUB/SUB ──
PUBLISH    notifications:user:123 '{"type":"order_shipped","id":456}'
SUBSCRIBE  notifications:user:123
PSUBSCRIBE notifications:user:*      # pattern matching

── STREAMS ──
# Produce
XADD orders * orderId 456 userId 123 status shipped
# * = auto-generate ID (millisecond-sequence)

# Consume from beginning
XREAD COUNT 10 STREAMS orders 0

# Block for new messages
XREAD BLOCK 0 STREAMS orders $

# Consumer Groups: multiple parallel consumers, each gets unique messages
XGROUP CREATE orders email-svc $          # create group, start from now
XREADGROUP GROUP email-svc consumer1 COUNT 10 STREAMS orders >
XACK orders email-svc <message-id>        # confirm processed
XPENDING orders email-svc - + 10        # see unacknowledged messages
Pub/Sub vs Streams choice

Pub/Sub is a broadcast fire-and-forget — if the subscriber is offline, the message is lost. Use it only for ephemeral real-time data: live notifications, presence updates, live dashboards. Streams are the modern default for most messaging use cases — messages are persisted, consumer groups enable parallel processing with at-least-once delivery via XACK.

Quick recall

Pub/Sub = ephemeral broadcast, fire-and-forget, no persistence, all subscribers get all messages. Streams = persistent log, consumer groups for parallel workers, XACK for at-least-once delivery, replay via message IDs. Streams ≈ Redis's Kafka equivalent for simpler use cases.

T10

Pipelining & batching — eliminating round-trip latency

Without pipelining, each command waits for a network round-trip (RTT) before the next command sends. With pipelining, you send N commands without waiting for responses, then read all N responses at once. Result: N×RTT reduced to ~1 RTT.

javascript — Pipeline with ioredis
// Without pipeline: 100 round-trips ≈ 50ms on localhost
for (const key of keys) {
  await redis.get(key);  // each waits for response
}

// With pipeline: 1 round-trip ≈ 0.5ms
const pipeline = redis.pipeline();
for (const key of keys) {
  pipeline.get(key);    // queued locally
}
const results = await pipeline.exec();  // single round-trip

// Pipeline vs MULTI/EXEC:
// Pipeline: commands sent together, server may interleave with other clients
// MULTI/EXEC: atomic — server buffers then executes without interruption

// SCAN — safe key enumeration (NEVER use KEYS * in production!)
SCAN 0 MATCH user:* COUNT 100   # cursor-based, non-blocking
# returns: [nextCursor, [matching_keys]]
# iterate until cursor returns 0
# KEYS * blocks Redis until complete — do not use in production!
Never use KEYS * in production

KEYS * blocks the entire Redis event loop until it scans the full keyspace — on a 10M-key instance this can take seconds, blocking all other clients. Use SCAN with a cursor instead: it's iterative, returns a small batch per call, and never blocks.

Quick recall

Pipelining = send N commands without waiting, collect N responses in one trip. N×RTT → ~1 RTT. Pipeline is client-side batching — no atomicity guarantee (use MULTI/EXEC for that). SCAN vs KEYS: SCAN is cursor-based and non-blocking; KEYS blocks. HSCAN/SSCAN/ZSCAN for type-specific iteration.

T11 Advanced

Redis Sentinel — automatic failover and master election

Sentinel provides high availability for a single Redis primary + replicas. It monitors the master, detects failures, elects a new master from replicas, and notifies clients of the new address — all automatically.

text — Sentinel architecture
[ Sentinel 1 ]   [ Sentinel 2 ]   [ Sentinel 3 ]
      │                 │                 │
      └─────────── monitors via PING ──────┘
                        │
             ┌──────────┴──────────┐
      [ Master Redis ]    [ Replica 1 ]   [ Replica 2 ]
             │                    ↑
             └──── replication ───┘

Failover sequence when master goes down:
1. Sentinels detect SDOWN (subjectively down) via PING timeout
2. Quorum agreement: 2/3 sentinels agree → ODOWN (objectively down)
3. Sentinel leader elected (Raft-like)
4. Best replica promoted to master (lowest replication lag wins)
5. Other replicas reconfigured to replicate from new master
6. Clients notified via Sentinel API
config + javascript — Sentinel setup
# sentinel.conf
sentinel monitor mymaster 127.0.0.1 6379 2
# "mymaster" = name, quorum = 2 (2 sentinels must agree on failure)
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1   # how many replicas sync at once

// Node.js ioredis: connect via Sentinel, not master directly
const redis = new Redis({
  sentinels: [
    { host: 'sentinel1', port: 26379 },
    { host: 'sentinel2', port: 26379 },
    { host: 'sentinel3', port: 26379 }
  ],
  name: 'mymaster'   // Sentinel resolves current master address
});
Sentinel in practice

Minimum 3 Sentinel nodes (odd number for quorum). Sentinels must run on separate machines from Redis for meaningful HA. In cloud environments (AWS ElastiCache, GCP Memorystore), HA is managed automatically — prefer managed Redis over self-managed Sentinel unless you have a specific reason.

Quick recall

Sentinel = HA for single primary + replicas. Quorum (usually 2/3) must agree before failover. Clients connect to Sentinel, not master directly — Sentinel resolves current master. 3 Sentinels minimum. Failover time: ~30 seconds default (down-after-milliseconds + election + sync).

T12 Advanced

Redis Cluster — horizontal scaling with hash slots

Redis Cluster shards data automatically across multiple master nodes using 16,384 hash slots. Each master owns a range of slots. Keys are mapped via CRC16(key) % 16384. This enables scaling beyond single-node memory limits.

text — Cluster architecture and slot routing
Redis Cluster = 16384 hash slots distributed across masters

[ Master A ]  slots    0–5460    +  [ Replica A ]
[ Master B ]  slots 5461–10922   +  [ Replica B ]
[ Master C ]  slots 10923–16383  +  [ Replica C ]

Slot assignment: CRC16(key) % 16384
  SET user:123  →  CRC16("user:123") % 16384  →  slot 5298  →  Master A

Client connects to any node:
  If wrong node: MOVED 5298 127.0.0.1:6379
  Client redirects to correct node (caches slot map locally)
redis-cli — Hash tags for multi-key operations
# Multi-key commands (MGET, MSET, SUNION) only work if all keys are in same slot
# Hash tags: {…} portion used for slot calculation

# These two keys will be on the same slot (hash tag = "user:123")
SET {user:123}:profile '...'
SET {user:123}:settings '...'
MGET {user:123}:profile {user:123}:settings   # works!

# Cross-slot transactions are not supported
# MULTI/EXEC fails if commands span multiple slots

# Cluster management
CLUSTER INFO
CLUSTER NODES
CLUSTER KEYSLOT user:123    # which slot does this key belong to?
Cluster limitations

Multi-key operations only work if all keys hash to the same slot. Use hash tags {} to force key colocation. No cross-slot transactions. Minimum 3 masters + 3 replicas for production. Lua scripts cannot access keys across multiple slots.

Quick recall

16384 hash slots, each master owns a range. CRC16(key) % 16384 = slot. MOVED response = client redirect to correct node. Hash tags {} = force colocation on same slot. No cross-slot multi-key ops. Minimum 6 nodes (3 masters + 3 replicas) for production cluster.

T13 Core

Caching patterns — Cache-Aside, Write-Through, Write-Behind

PatternHow it worksConsistencyBest for
Cache-AsideApp checks cache; miss → fetch DB → populate cacheEventual (stale on write until TTL expires)Read-heavy workloads (most common)
Write-ThroughEvery write updates cache AND DB synchronouslyStrongRead-heavy + consistency critical
Write-BehindWrite to cache first; async write to DB laterWeak (loss risk if Redis crashes)Write-heavy, tolerable data loss
Read-ThroughCache handles miss automatically (cache populates itself)EventualWhen cache layer manages its own loading
javascript — Cache-Aside pattern (Node.js)
async function getUser(userId) {
  const cacheKey = `user:${userId}`;

  // 1. Check cache first
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — fetch from DB
  const user = await db.query('SELECT * FROM users WHERE id = ?', [userId]);

  // 3. Populate cache with TTL
  await redis.setex(cacheKey, 3600, JSON.stringify(user));
  return user;
}

// Cache invalidation on update
async function updateUser(userId, data) {
  await db.query('UPDATE users SET ...', [userId]);
  await redis.del(`user:${userId}`);  // invalidate — next read re-populates
}

// Write-Through: update cache AND DB synchronously
async function updateUserWT(userId, data) {
  const updated = await db.query('UPDATE users ...', [userId]);
  await redis.setex(`user:${userId}`, 3600, JSON.stringify(updated));
}
Cache invalidation is the hard part

Cache-Aside is the most common and safest pattern — only cache what's actually requested (no cold data waste). The tricky part is invalidation: delete on update, use TTL as a safety net. Write-Through adds write latency but keeps the cache consistent immediately. Write-Behind is fastest for writes but risks data loss if Redis crashes before the async DB write.

Quick recall

Cache-Aside = check cache → miss → DB → populate (most common). Write-Through = write cache + DB together (consistent). Write-Behind = write cache first + async DB (fast writes, loss risk). Always set a TTL as a safety net against stale data even when doing explicit invalidation.

T14 Advanced

Cache stampede — thundering herd problem and prevention strategies

A popular cache key expires → thousands of concurrent requests simultaneously miss → all query the DB → DB overloads → latency spikes → cascade failure. This is the thundering herd or cache stampede.

javascript — Prevention strategies
── Strategy 1: Mutex lock (most reliable) ──
async function getCachedData(key) {
  let data = await redis.get(key);
  if (data) return JSON.parse(data);

  // Try to acquire lock: SET NX with 10s auto-expire
  const lockAcquired = await redis.set(`lock:${key}`, '1', 'NX', 'EX', 10);

  if (lockAcquired) {
    // Only this request fetches from DB
    const fresh = await fetchFromDB(key);
    await redis.setex(key, 3600, JSON.stringify(fresh));
    await redis.del(`lock:${key}`);
    return fresh;
  } else {
    // Others wait ~50ms then hit the now-populated cache
    await sleep(50);
    return getCachedData(key);  // retry
  }
}

── Strategy 2: TTL jitter (prevents mass expiry) ──
// Add random offset so bulk-loaded keys don't expire simultaneously
const ttl = 3600 + Math.floor(Math.random() * 300);  // 3600±300s
await redis.setex(key, ttl, data);

── Strategy 3: Probabilistic early recompute ──
// Start refreshing randomly before expiry — no stampede possible
const timeLeft = await redis.pttl(key);   // ms remaining
const shouldRefresh = Math.random() < Math.exp(-0.5 * timeLeft / 1000);
TTL jitter is mandatory for bulk loads

If you populate 10,000 product cache entries all with the same TTL of 3600 seconds, they all expire at the same moment. Add ±10% random jitter to every TTL when bulk-loading. This one line prevents a guaranteed stampede 60 minutes after every cache warm-up.

Quick recall

Stampede = mass cache miss → DB overload. Mutex = only first request fetches, others wait then hit warm cache. TTL jitter = ±10% random offset prevents synchronized expiry. Probabilistic early recompute = refresh randomly before expiry to prevent mass expiry entirely. Use all three for truly high-traffic keys.

T15 Advanced

Distributed locking — correct Redis lock implementation and Redlock

A distributed lock ensures only one process executes a critical section at a time across multiple servers. The naive implementation has a race condition; the correct pattern uses a unique value, atomic SET NX, and Lua for safe release.

javascript — Single-node lock and Redlock
── Correct single-node Redis lock ──
const lockId = crypto.randomUUID();  // unique value = owner identity

const acquired = await redis.set(
  'lock:payment:order123',
  lockId,
  'NX',    // set only if not exists
  'EX', 30 // auto-expire in 30s — prevents deadlock if process crashes
);

if (!acquired) throw new Error('Could not acquire lock');

try {
  await processPayment('order123');
} finally {
  // MUST use Lua: atomically check value then delete
  // Two separate commands (GET + DEL) would have a race condition!
  await redis.eval(`
    if redis.call('get', KEYS[1]) == ARGV[1] then
      return redis.call('del', KEYS[1])
    else return 0 end
  `, 1, 'lock:payment:order123', lockId);
}

── Redlock: distributed lock across N independent Redis nodes ──
// Use the redlock npm library; acquire lock from majority (N/2+1) nodes
const redlock = new Redlock([redis1, redis2, redis3]);
const lock = await redlock.acquire(['lock:resource'], 30000);
try {
  await doWork();
} finally {
  await lock.release();
}
Critical mistake to avoid

Never use GET then DEL to release a lock — another process could steal and delete your lock between the two commands. Always use the Lua script that atomically checks the value matches your lockId before deleting. The TTL auto-expiry is your deadlock safety net — if the process crashes, the lock auto-releases.

Quick recall

Lock acquire: SET NX EX (atomic). Lock release: Lua script (check value == lockId, then DEL). Never GET+DEL — race condition. Lock value = UUID (proves ownership). TTL = deadlock prevention. Redlock = acquire from majority of N independent nodes; safer when one Redis node can fail mid-lock.

T16

Rate limiting with Redis — fixed window, sliding window, token bucket

javascript — Rate limiting implementations
── Fixed Window (simplest) ──
async function fixedWindowLimit(userId, limit=100, window=60) {
  const key = `ratelimit:${userId}:${Math.floor(Date.now()/1000/window)}`;
  const count = await redis.incr(key);
  if (count === 1) await redis.expire(key, window);
  return count <= limit;
}
// Weakness: burst at window boundary — 100 at 11:59, 100 at 12:00 = 200 in 2s

── Sliding Window Log (accurate, more memory) ──
async function slidingWindowLog(userId, limit=100, window=60) {
  const key = `ratelimit:log:${userId}`;
  const now = Date.now();
  const cutoff = now - (window * 1000);

  await redis.pipeline()
    .zremrangebyscore(key, '-inf', cutoff)  // remove old timestamps
    .zadd(key, now, `${now}-${Math.random()}`)
    .expire(key, window)
    .exec();

  const count = await redis.zcard(key);
  return count <= limit;
}
// Uses Sorted Set: score = timestamp, member = unique request ID
// Remove entries older than window, check count after adding current

Fixed windowVulnerable to boundary burstO(1)No

Production recommendation

The sliding window counter (weighted blend of current + previous window) implemented in Lua is both accurate and O(1) memory. Use it for API rate limiting. For burst-tolerant rate limits (download quotas, batch jobs), token bucket is more appropriate. Always implement in Lua to avoid INCR+EXPIRE race conditions.

Quick recall

Fixed window = O(1) but boundary burst vulnerability. Sliding window log = exact but O(N) memory per user. Sliding window counter = O(1) and ~accurate (weighted blend). Token bucket = O(1) with controlled burst. All production implementations should use Lua for atomicity to avoid INCR+EXPIRE race conditions.

T17

Session management with Redis — distributed sessions, TTL sliding, revocation

Redis is the standard choice for session storage in distributed Node.js applications. Any server can look up any session, enabling horizontal scaling without sticky sessions or shared database locks.

javascript — Express session with Redis
const session = require('express-session');
const RedisStore = require('connect-redis')(session);

app.use(session({
  store: new RedisStore({
    client: redisClient,
    prefix: 'sess:',  // key prefix: sess:{sessionId}
    ttl: 86400         // 24h TTL
  }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: {
    secure: true,       // HTTPS only
    httpOnly: true,      // no JS access (XSS protection)
    maxAge: 86400000,
    sameSite: 'strict'  // CSRF protection
  }
}));

// Slide TTL on each request (keep active users logged in)
app.use((req, res, next) => {
  if (req.session.userId) redis.expire(`sess:${req.sessionID}`, 86400);
  next();
});

// Logout: delete session key → immediate revocation
req.session.destroy(async () => {
  await redis.del(`sess:${req.sessionID}`);
});

// Force logout from all devices for a user (store session IDs per user)
const sessionIds = await redis.smembers(`user-sessions:${userId}`);
await Promise.all(sessionIds.map(id => redis.del(`sess:${id}`)));
Redis sessions vs JWT

The key advantage of Redis sessions over stateless JWT: immediate revocation. Delete the session key and the user is logged out instantly. JWT tokens live until they expire (potentially hours). For security-sensitive apps (banking, admin panels), Redis sessions are the correct choice.

Quick recall

Redis sessions = any server can look up any session (no sticky sessions needed). Sliding TTL = reset expire on each request (active users stay logged in). Immediate revocation = delete key → logged out (vs JWT where you wait for expiry). Store session IDs per user in a Set to enable "logout all devices".

T18

Redis as a message queue — Lists, Streams, and BullMQ

javascript — Queue patterns
── Simple queue with Lists ──
// Producer: LPUSH jobs to head
await redis.lpush('queue:emails', JSON.stringify({
  to: 'user@mail.com', subject: 'Welcome!'
}));

// Consumer: blocking pop from tail (FIFO)
while (true) {
  const [, job] = await redis.brpop('queue:emails', 0);
  await sendEmail(JSON.parse(job));
}

── Reliable queue: LMOVE prevents message loss on crash ──
// Atomically move from queue to processing list
const job = await redis.lmove(
  'queue:emails',       // source
  'processing:emails',  // destination (in-flight tracking)
  'RIGHT', 'LEFT'
);
await processJob(job);
await redis.lrem('processing:emails', 1, job);  // remove when done
// If crash mid-processing: job stays in processing list for requeue

── BullMQ (production Redis queue library) ──
const queue = new Queue('emails', { connection: redis });
await queue.add('send', { to: 'user@mail.com' }, {
  attempts: 3,          // retry 3 times on failure
  backoff: { type: 'exponential', delay: 1000 },
  removeOnComplete: 100, // keep last 100 completed jobs
  delay: 60000           // delayed job: run after 60s
});
When Redis queue vs Kafka

Use Redis queues (BullMQ) when: moderate volume (thousands/sec, not millions), no event replay needed, you want simple ops and a Node.js ecosystem. Upgrade to Kafka when: millions of events/sec, you need event replay/audit log, or multiple independent consumer teams need to read the same stream.

Quick recall

LPUSH + BRPOP = simple FIFO queue. LMOVE to processing list = reliable queue (no message loss on crash). Streams with consumer groups = at-least-once delivery with XACK. BullMQ = production abstraction: retries, backoff, delayed jobs, priority, rate limiting. Redis queue ≠ Kafka — no replay, simpler ops model.

Security best practices — auth, ACLs, TLS, network isolation

The most dangerous Redis misconfiguration is binding to 0.0.0.0 with no password. Redis is frequently targeted by automated scanners; an exposed instance can be wiped with a single FLUSHALL command.

config — redis.conf security hardening
# 1. Bind to specific interfaces only
bind 127.0.0.1 10.0.0.5       # localhost + private IP only, not 0.0.0.0!

# 2. Require password (Redis 5 and below)
requirepass your_strong_password_here

# 3. Protected mode (refuse external connections if no auth configured)
protected-mode yes

# 4. Disable dangerous commands (can't be re-enabled at runtime)
rename-command FLUSHALL  ""   # disables FLUSHALL entirely
rename-command FLUSHDB   ""
rename-command CONFIG    ""   # prevents runtime config changes
rename-command DEBUG     ""

# 5. Enable TLS (Redis 6+)
tls-port 6380
tls-cert-file /etc/ssl/redis/redis.crt
tls-key-file  /etc/ssl/redis/redis.key
tls-ca-cert-file /etc/ssl/redis/ca.crt
redis-cli — ACLs (Redis 6+)
# Principle of least privilege via ACL
# App user: only GET/SET/DEL/EXPIRE on cache:* keys
ACL SETUSER appuser on >password ~cache:* +GET +SET +DEL +EXPIRE

# Read-only analytics user
ACL SETUSER readonly on >ropass ~* +GET +MGET +HGET +HGETALL +LRANGE +ZRANGE

# Disable default user
ACL SETUSER default off

# List all ACL users
ACL LIST
ACL WHOAMI
Non-negotiable in production

Use security groups / firewall rules to allow only app servers to reach the Redis port (6379) — no public internet access. Even with a password, a publicly accessible Redis is a security risk. Always use ACLs in Redis 6+ to give each service the minimum required permissions. Disable FLUSHALL and FLUSHDB.

Quick recall

Bind to private IPs only. requirepass or ACL auth. Protected-mode = yes. Disable FLUSHALL/FLUSHDB/CONFIG/DEBUG via rename-command. TLS in Redis 6+. ACLs for per-user key pattern restrictions and command restrictions. Security group rules = only app servers can reach Redis port. Never expose Redis to public internet.
