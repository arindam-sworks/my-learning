Chapter 1 — Indexing & Performance
T1

Clustered vs non-clustered index in InnoDB

AspectClustered IndexSecondary (Non-Clustered)
What is it?Row data physically stored in PK orderSeparate B-tree; leaf holds Primary Key value
Per tableExactly one (the PRIMARY KEY)Up to 64 per table
Leaf page holdsActual row dataPrimary Key value only → double-read
Range queriesVery fastExtra PK lookup required
InnoDB detail

In InnoDB, the PRIMARY KEY is the clustered index. If you define no PK, MySQL creates a hidden rowid. Secondary index lookups become: index scan + PK lookup (two reads).

Quick recall

Clustered = data sorted in PK order. Only one per table. PK = Clustered. All others are Secondary (non-clustered).

T2

Composite index (a, b, c) — which queries use it?

Composite index = one index across multiple columns. The key rule is the Leftmost Prefix Rule.

QueryIndex used?Reason
WHERE a = 1✅ YesLeftmost prefix match
WHERE a = 1 AND b = 2✅ Yesa, b prefix match
WHERE a = 1 AND b = 2 AND c = 3✅ YesFull match
WHERE b = 2❌ NoSkipped leftmost column a
WHERE a = 1 AND c = 3⚠ PartialOnly uses a; c skipped
WHERE a LIKE 'x%' AND b = 2⚠ Range on aAfter a range, subsequent columns not usable

Quick recall

Start from the left, never skip. A range condition (LIKE, >, <) stops further column use in the same index.

T3

Index cardinality and the query optimiser

Cardinality = the number of distinct values in an indexed column (an estimate, not exact).

  • High cardinality (e.g. user_id, email) = highly selective → index is very effective
  • Low cardinality (e.g. gender, status = 'active/inactive') = less selective → full scan may be faster
  • The optimiser reads statistics from INFORMATION_SCHEMA.STATISTICS and uses cardinality to decide whether to use an index
  • Run ANALYZE TABLE to refresh statistics — outdated stats lead to wrong query plans

Quick recall

Indexing a low-cardinality column is often wasted — the optimiser ignores it. In composite indexes, place high-cardinality columns first.

T4

EXPLAIN output: type: ALL — what it means and how to fix it

type: ALL = full table scan. On a 10M-row table, this is a disaster for latency.

EXPLAIN type hierarchy (best → worst)
system > const > eq_ref > ref > range > index > ALL

-- system:  single-row table (constant)
-- const:   PK/UNIQUE equality — at most one row
-- eq_ref:  PK match in a JOIN
-- ref:     non-unique index equality
-- range:   index range scan (BETWEEN, IN, >, <)
-- index:   full index scan (better than ALL — index is smaller)
-- ALL:     full table scan ← investigate if you see this

How to fix:

  • Verify that columns in WHERE / JOIN conditions are indexed
  • Avoid wrapping indexed columns in functions: WHERE YEAR(created_at) = 2024 breaks index use. Use WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31' instead
  • Avoid implicit type casts — comparing a string column to an integer causes a cast that bypasses the index
  • FORCE INDEX hint is available as a last resort

Quick recall

Always run EXPLAIN before optimising. type = ALL + rows = millions = red flag requiring immediate attention.

T5

When do indexes hurt performance?

  • Write-heavy tables: every INSERT / UPDATE / DELETE must update all indexes. An order-processing table with many indexes slows writes significantly.
  • Low-cardinality columns: an index on is_deleted (true/false) is often ignored by the optimiser — full scan can be faster.
  • Small tables: the index overhead exceeds the benefit on a 100-row table.
  • Unused indexes: even if no query uses them, they consume storage and slow every write. Identify and drop unused indexes via INFORMATION_SCHEMA.
  • Over-indexing: adding an index to every column is a common startup mistake; write speed drops dramatically.

Quick recall

Indexes speed reads and tax writes. Read-heavy workload → more indexes. Write-heavy workload → fewer, more targeted indexes.

Chapter 2 — Transactions & Locking
T6

Four transaction isolation levels — anomalies prevented

LevelDirty ReadNon-Repeatable ReadPhantom Read
READ UNCOMMITTED❌ occurs❌ occurs❌ occurs
READ COMMITTED✅ prevented❌ occurs❌ occurs
REPEATABLE READ ← default✅ prevented✅ prevented⚠ mostly prevented*
SERIALIZABLE✅ prevented✅ prevented✅ prevented

* InnoDB REPEATABLE READ uses MVCC to prevent phantom reads in most cases — but not in SELECT ... FOR UPDATE scenarios.

  • Dirty Read: reading uncommitted data — if the other transaction rolls back, you saw wrong data
  • Non-Repeatable Read: same row read twice in one transaction yields different values
  • Phantom Read: a range query run twice sees new rows inserted by another transaction

Quick recall

Default = REPEATABLE READ. This is correct for most production applications. Use READ COMMITTED for audit logs where you need the latest committed data.

T7

Deadlocks — scenario and prevention

Two transactions wait for each other’s locks indefinitely. MySQL automatically rolls back one of them.

Deadlock Scenario
-- Transaction 1                     Transaction 2
START TRANSACTION;                   START TRANSACTION;
UPDATE accounts SET bal=bal-100      UPDATE accounts SET bal=bal-200
WHERE id = 1;  ← locks row 1        WHERE id = 2;  ← locks row 2

UPDATE accounts SET bal=bal+100      UPDATE accounts SET bal=bal+100
WHERE id = 2;  ← waiting for T2!    WHERE id = 1;  ← waiting for T1!
-- DEADLOCK! MySQL kills one transaction.

Prevention strategies:

  • Always lock rows in the same order (e.g. ascending id) — if all transactions follow the same sequence, circular waits cannot form
  • Keep transactions short — hold locks for as little time as possible
  • Implement retry logic in the application — catch the deadlock error and retry
  • Use SELECT ... FOR UPDATE only when genuinely needed
Diagnose

Run SHOW ENGINE INNODB STATUS\G and look at the LATEST DETECTED DEADLOCK section for the exact lock chain.

Quick recall

Deadlock = circular wait. Break the cycle by enforcing a consistent lock-acquisition order across all code paths.

T8

Shared lock vs exclusive lock — and SELECT ... FOR UPDATE

Shared Lock (S)Exclusive Lock (X)
Acquired bySELECT ... LOCK IN SHARE MODEINSERT/UPDATE/DELETE or SELECT ... FOR UPDATE
Compatible with other S✅ Yes❌ No
Compatible with X❌ No❌ No
Use case“I’m reading — don’t delete this row”“I’m about to mutate — nobody touch this row”
SELECT ... FOR UPDATE example
START TRANSACTION;

-- Acquire exclusive row lock; other transactions cannot update this row
SELECT * FROM inventory
WHERE product_id = 42
FOR UPDATE;

-- Now safely decrement stock — no concurrent update possible
UPDATE inventory
SET stock = stock - 1
WHERE product_id = 42;

COMMIT;

Quick recall

FOR UPDATE = “I will mutate this row — lock it exclusively until commit.” Essential for inventory decrement, bank balance update, and similar read-modify-write patterns.

T9

MVCC — Multi-Version Concurrency Control

MVCC keeps multiple row versions. Readers never block writers; writers never block readers. “Readers don’t block writers, writers don’t block readers.”

  • Every row carries two hidden columns: DB_TRX_ID (which transaction created this version) and DB_ROLL_PTR (pointer to the undo log chain)
  • When a transaction reads, it uses a read view to see only the row version visible at transaction start
  • An UPDATE creates a new version; the old version moves to the undo log
  • The purge thread eventually cleans old versions when no active transaction needs them
Production impact

A long-running transaction prevents old versions from being purged — the undo log grows and performance degrades. Always commit transactions quickly.

Quick recall

MVCC = snapshot. You see the database as it was when your transaction started. Concurrent changes do not affect your view (under REPEATABLE READ).

T23 ✦ New

ACID — what each property means at the InnoDB level

The textbook definition is easy. Senior interviewers want to know how InnoDB actually implements each guarantee — not just what the acronym stands for.

PropertyInnoDB mechanismWhat breaks it
AtomicityUndo log — every change is recorded so the entire transaction can be rolled back atomicallyApplication crash mid-transaction (InnoDB auto-rolls back on reconnect)
ConsistencyConstraints enforced at commit: UNIQUE, NOT NULL, FK, CHECKDisabling FK checks (SET FOREIGN_KEY_CHECKS=0), bypassing constraints
IsolationMVCC for reads + row-level locks for writes, controlled by isolation levelChoosing too-low isolation (READ UNCOMMITTED) exposes dirty reads
DurabilityRedo log (WAL) flushed to disk before the commit ACK is returned to the clientinnodb_flush_log_at_trx_commit=2 — data in OS cache, lost on OS crash
The WAL angle interviewers love

WAL = Write-Ahead Log. InnoDB writes the change to the redo log on disk before touching the actual data page (which can be flushed lazily). On a crash, MySQL replays the redo log for committed transactions and uses the undo log to discard uncommitted ones. This is what makes durability possible without syncing every data page to disk on every commit. Setting innodb_flush_log_at_trx_commit=1 is the only fully ACID-safe value.

Quick recall

A = undo log (rollback). C = constraints at commit. I = MVCC + locks. D = redo log (WAL) flushed before ACK. innodb_flush_log_at_trx_commit=1 = fully durable.

T24 ✦ New

Gap locks and next-key locks — the hidden cause of range deadlocks

Under REPEATABLE READ, InnoDB locks more than just the rows you touch. Understanding this prevents unexpected deadlocks in production that are impossible to reproduce with a single row.

Lock typeWhat it locksWhen it appears
Record lockA single existing index rowAny WHERE clause that hits a unique index with an exact match
Gap lockThe gap between index values (no row, just space)Range queries (BETWEEN, <, >) on non-unique indexes under REPEATABLE READ
Next-key lockRecord lock + gap lock to the leftDefault for most InnoDB row locking under REPEATABLE READ
SQL — Gap lock scenario
-- Table: orders(id PK, status, amount)
-- Index on status

-- Tx A — scans all 'pending' rows
SELECT * FROM orders
WHERE status = 'pending'
FOR UPDATE;

-- Tx B — tries to INSERT a new pending order → blocked!
-- Tx A holds a gap lock on the 'pending' gap.
-- Tx B must wait until Tx A commits or rolls back.
INSERT INTO orders (status, amount)
VALUES ('pending', 500);
How to reduce gap lock contention
  • Use READ COMMITTED isolation — gap locks are disabled (dirty reads allowed; weaker guarantee)
  • Query by Primary Key when possible — a PK exact-match uses a record lock only, no gap
  • Narrow range queries: fewer gaps locked = less contention
  • Keep transactions short — the lock is released on commit

Quick recall

Gap lock = InnoDB locks the space between index values, not just the row. Prevents phantom reads. Exists only under REPEATABLE READ. A range FOR UPDATE blocks inserts into that range. Switch to READ COMMITTED to eliminate gap locks.

T25 ✦ New

Optimistic vs pessimistic locking — when to use each

Both patterns solve the lost-update problem. The choice affects throughput, contention, and complexity at the application layer.

Pessimistic lockingOptimistic locking
MechanismSELECT ... FOR UPDATE — row locked immediatelyversion column — lock-free read; check version on update
When it blocksAt read time — other writers queue upAt write time — transaction retried if version mismatch
Best forHigh-contention resources (e.g. inventory, seat bookings, bank balance)Low-contention reads where conflicts are rare (e.g. user profile edit)
ThroughputLower — lock held for entire transaction durationHigher — no lock held during read phase
Failure modeDeadlock if lock ordering is wrongRetry storm if contention is actually high
Pessimistic — SELECT FOR UPDATE
START TRANSACTION;

-- Lock the row immediately; other writers wait
SELECT stock
FROM   products
WHERE  id = 42
FOR UPDATE;

UPDATE products
SET    stock = stock - 1
WHERE  id = 42;

COMMIT;
Optimistic — version column pattern
-- Read without any lock
SELECT id, stock, version
FROM   products
WHERE  id = 42;

-- Update only if version has not changed since we read it
UPDATE products
SET    stock   = stock - 1,
       version = version + 1
WHERE  id = 42
  AND  version = 7;   -- 7 = value we read earlier

-- If rows_affected = 0 → someone else updated; retry the whole flow
Interview answer structure

State which pattern you would choose and why for the given scenario. E.g.: “For a flash-sale inventory counter with 500 concurrent buyers, I would use pessimistic locking — the contention is guaranteed and retries would create a stampede. For a CMS where two editors might rarely touch the same article, optimistic locking is better — no lock overhead and conflicts surface as a clear error to the user.”

Quick recall

Pessimistic = lock at read (FOR UPDATE). Optimistic = check version at write. Pessimistic for high contention; Optimistic for low contention. Optimistic retry storms kill you if you chose wrong.

Chapter 3 — Schema Design & Normalisation
T10

1NF, 2NF, 3NF — and when to denormalise in a startup

Normal FormRuleExample Violation
1NFAtomic values; no repeating groupsphones = "01711, 01811" (comma-separated in one cell)
2NF1NF + no partial dependency on part of a composite PKorder_items(order_id, product_id, product_name) — product_name depends only on product_id
3NF2NF + no transitive dependencyemployees(emp_id, dept_id, dept_name) — dept_name depends on dept_id, not directly on emp_id

When to denormalise in a startup:

  • A dashboard or report query joins 5+ tables and runs slowly → denormalise into a pre-aggregated summary table
  • Read-to-write ratio is 100:1 or higher → some redundancy is acceptable
  • Use a separate analytics DB or materialised view rather than denormalising the transactional DB
  • Use JSON columns for flexible attributes (MySQL 5.7+) instead of wide denormalised tables

Quick recall

Normalise for data integrity. Denormalise for read speed. In production: design 3NF first, then denormalise only where you have measured a bottleneck.

T26 ✦ New

Foreign key constraints — benefits, tradeoffs, and the startup debate

Every senior engineer is asked this. The answer is not “always use them” or “never use them” — it is “here is the tradeoff and here is when I choose each.”

With FK constraintsWithout FK constraints (app-level)
Data integrityEnforced at DB level — orphan rows impossibleDepends entirely on application code
Write performanceEvery INSERT/UPDATE/DELETE checks parent row — extra index lookupNo overhead; pure write throughput
Schema migrationMust drop FK before altering referenced table; more complex DDLAlter freely without worrying about FK order
Multi-shard / microservicesCannot enforce FK across databases or servicesApplication-level consistency checks are the only option
DebuggingDB tells you exactly what violated the constraintSilent data corruption — discovered later via reports
Common senior answer

Use FK constraints in monolithic relational schemas where all tables live in the same MySQL instance and data integrity is paramount (finance, healthcare, e-commerce orders). Drop FK constraints when you need maximum write throughput (event ingestion, logging), when tables are across different databases (sharding, microservices), or during bulk data loads (SET FOREIGN_KEY_CHECKS=0, re-enable and validate after load).

SQL — FK with cascade options
CREATE TABLE orders (
  id          INT PRIMARY KEY,
  customer_id INT NOT NULL,
  -- ON DELETE CASCADE: delete orders when customer deleted
  -- ON DELETE RESTRICT: block customer deletion if orders exist
  -- ON DELETE SET NULL: orphan the row (customer_id = NULL)
  FOREIGN KEY (customer_id)
    REFERENCES customers(id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE
);
The bulk load pattern

When importing millions of rows, disable FK checks for the session, load the data, then re-enable and run a validation query. This is standard ETL practice — never leave FK checks off permanently.

Quick recall

FK = DB-level integrity vs write overhead. Use in monolithic OLTP, skip across shards/services. CASCADE deletes child rows; RESTRICT prevents deletion; SET NULL orphans. Disable during bulk loads, validate after.

T14

WHERE vs HAVING — difference and performance impact

WHEREHAVING
Executes whenBefore GROUP BYAfter GROUP BY
FiltersIndividual rowsGrouped results
Aggregate functions❌ Not allowed✅ Allowed (COUNT, SUM, etc.)
Index used✅ Yes❌ No — runs after aggregation
Performance comparison
-- ❌ SLOW: groups all rows first, then filters
SELECT dept_id, COUNT(*) FROM employees
GROUP BY dept_id
HAVING dept_id = 5;   -- dept_id is not an aggregate — belongs in WHERE

-- ✅ FAST: filter first (uses index), then group a smaller set
SELECT dept_id, COUNT(*) FROM employees
WHERE dept_id = 5
GROUP BY dept_id;

Quick recall

The earlier you filter, the less data gets processed. Non-aggregate condition → WHERE. Aggregate condition (e.g. COUNT(*) > 5) → HAVING.

T15

Slow JOIN on 5M-row tables — systematic optimisation

  • Step 1 — Run EXPLAIN: check type, key, rows. Which table is being full-scanned?
  • Step 2 — Index join columns: ON t1.user_id = t2.user_id — both sides need indexes
  • Step 3 — Filter selectivity: is the WHERE condition on an indexed column? Push filters as early as possible to reduce row count before the join
  • Step 4 — Join order: MySQL usually drives from the smaller filtered table. Use STRAIGHT_JOIN to force a specific order if the optimiser chooses wrong
  • Step 5 — Covering index: if all SELECTed columns are in the index, table lookups are eliminated entirely
  • Step 6 — Query rewrite: convert correlated subqueries to JOINs. Compare EXISTS vs IN — test which the optimiser handles better for your data distribution
  • Step 7 — Keyset pagination: large OFFSET values force MySQL to scan and discard. Use WHERE id > last_seen_id LIMIT 20 instead

Quick recall

In interviews, walk through this framework step by step — it shows systematic thinking, not just “add an index.”

T27 ✦ New

IN vs EXISTS vs JOIN — which to use and why

This is one of the most common senior SQL optimisation questions. The right answer depends on table sizes, nullability, and how the optimiser rewrites the query.

IN (subquery)EXISTSJOIN
Best whenSubquery result set is small; list is known at query timeSubquery table is large; early exit on first matchYou also need columns from the joined table; deduplication with DISTINCT
NULL behaviourDangerous: IN (1, NULL) — if subquery returns any NULL, the outer query returns no rows for non-matching IDsSafe with NULLs — returns TRUE/FALSE/FALSE (not NULL)INNER JOIN excludes NULLs naturally
Short-circuitNo — evaluates full subqueryYes — stops at first matchNo — full join evaluation
MySQL 8 optimiserOften rewrites as semi-join, identical to EXISTS in practiceSemi-join when possibleMost direct plan
SQL — IN vs EXISTS comparison
-- IN: good when the subquery list is short and has no NULLs
SELECT * FROM customers
WHERE id IN (SELECT customer_id FROM orders WHERE status = 'paid');

-- EXISTS: better when orders is large; stops on first match per customer
SELECT * FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.id
    AND o.status = 'paid'
);

-- NOT IN is BROKEN when subquery can return NULL — use NOT EXISTS instead
-- NOT IN (1, 2, NULL) → returns empty set for any outer value
SELECT * FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.id
);
The NULL trap in NOT IN

This is the most common real bug. If orders.customer_id can be NULL and you write WHERE c.id NOT IN (SELECT customer_id FROM orders), you get zero rows back because x NOT IN (..., NULL) is NULL (unknown), never TRUE. Always use NOT EXISTS instead of NOT IN when the subquery column is nullable.

Quick recall

IN = good for short lists. EXISTS = better for large right-side tables (short-circuits). JOIN when you need columns from both sides. Never use NOT IN on a nullable column — use NOT EXISTS.

T28 ✦ New

Slow Query Log — enabling, reading, and acting on it

The Slow Query Log is the primary production tool for finding queries that need optimisation. Senior engineers are expected to know how to enable and use it.

MySQL config — enable slow query log
-- In my.cnf / my.ini (permanent)
slow_query_log          = 1
slow_query_log_file     = /var/log/mysql/slow.log
long_query_time         = 1          -- log queries taking > 1 second
log_queries_not_using_indexes = 1    -- also log full-scan queries
min_examined_row_limit  = 100        -- skip trivial queries

-- At runtime (no restart needed)
SET GLOBAL slow_query_log          = ON;
SET GLOBAL long_query_time         = 0.5;  -- 500 ms threshold
SET GLOBAL log_queries_not_using_indexes = ON;
Shell — analyse with mysqldumpslow
# Top 10 slowest queries by total time spent
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# Top 10 by query count (frequently slow = high priority)
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# pt-query-digest: richer analysis with percentiles and examples
pt-query-digest /var/log/mysql/slow.log | head -100
Key metricWhat it meansAction
Query_timeWall-clock execution timeOver 1s on OLTP is a red flag
Lock_timeTime spent waiting for a lockHigh = contention; check transaction isolation and long-running transactions
Rows_examinedRows scannedMuch higher than Rows_sent = full scan, missing index
Rows_sentRows returned to clientLow rows_sent + high rows_examined = optimise the WHERE clause
Investigation workflow
  1. Enable slow query log with threshold 1s (or lower in staging)
  2. Run mysqldumpslow or pt-query-digest to find top offenders
  3. Take the worst query and run EXPLAIN on it
  4. Add or fix the index, rewrite the query, or both
  5. Verify improvement: re-run EXPLAIN, check rows and Extra

Quick recall

Slow query log + mysqldumpslow -s t -t 10 = find the worst queries fast. Rows_examined ≫ Rows_sent = missing index. Lock_time spike = transaction contention.

Chapter 5 — Replication, Scaling & Architecture
T16

Primary-Replica replication and read-after-write consistency

  • The primary accepts all writes and records changes in the binary log
  • Replicas read the binary log and apply changes — asynchronous by default
  • Routing read queries to replicas reduces primary load
Read-after-write lag problem
-- User updates their profile (write goes to Primary)
UPDATE users SET name = 'Rahul' WHERE id = 1;

-- Profile page loads immediately (read from Replica) ← lag!
SELECT name FROM users WHERE id = 1;  -- may still show the old name

Solutions:

  • For a user reading their own profile immediately after writing: read from Primary for that one request; use Replica for other users’ data
  • Sticky session: route the same user’s reads to Primary for a short window after their write
  • Semi-synchronous replication: the write ACK waits for at least one replica to confirm
  • Monitor lag with GTID-based replication metrics

Quick recall

“Write Primary, Read Replica” is the first scaling step for most products. Handling replication lag is the tricky design challenge.

T17

Sharding — when to do it and the challenges

Horizontal split: distribute a large table across multiple database servers by a shard key (e.g. user_id ranges or consistent hash).

When to shard:

  • A single server can no longer handle the write throughput
  • The table is so large that even the index no longer fits in memory
  • Vertical scaling (bigger machine) is no longer cost-effective or available

Challenges:

  • Cross-shard queries: JOINs across shards are not possible in SQL — data must be merged at the application layer
  • Rebalancing: adding a new shard requires migrating existing data, which is operationally complex
  • Distributed transactions: require Two-Phase Commit (2PC) — complex and slow
  • Hotspot: if many users land on the same shard (e.g. with a sequential ID key), the load is uneven

Quick recall

Sharding is a last resort. Exhaust caching, replication, indexing, and denormalisation first. Use consistent hashing to minimise data movement when rebalancing.

T18

Adding a column to a 100M-row table with zero downtime

Direct ALTER TABLE on a large table = table lock = downtime. Use online DDL instead.

  • pt-online-schema-change (Percona): creates a shadow table, syncs changes via triggers, then atomically renames — zero-downtime
  • gh-ost (GitHub): trigger-less; tracks changes from the binary log. Preferred in high-traffic production systems
  • MySQL 8+ Online DDL: ALGORITHM=INPLACE, LOCK=NONE supports many DDL operations without blocking — but not all change types qualify
MySQL Online DDL (verify support first)
ALTER TABLE big_table
ADD COLUMN new_col VARCHAR(100) DEFAULT NULL,
ALGORITHM=INPLACE,   -- modify in-place rather than creating a copy
LOCK=NONE;            -- no read/write lock during the operation

Quick recall

Production schema migration on large tables = gh-ost or pt-osc. Never run bare ALTER TABLE on a multi-million-row table in production without testing the DDL type first.

T29 ✦ New

Table partitioning — what it solves, when it does not help

Partitioning splits one logical table into separate physical segments. It can dramatically speed up time-series queries but is often misused as a substitute for proper indexing.

TypePartition keyBest for
RANGEDate / integer range (YEAR(created_at))Time-series data: logs, events, orders by year
LISTDiscrete values (region_id IN (1,2,3))Geo or category segregation
HASHHash of a columnEven data distribution; no range queries
KEYMySQL-controlled hash of PK/indexSimilar to HASH; uses internal algorithm
SQL — RANGE partitioning on a logs table
CREATE TABLE events (
  id         BIGINT AUTO_INCREMENT,
  created_at DATE NOT NULL,
  payload    JSON,
  PRIMARY KEY (id, created_at)   -- partition key MUST be in PK
)
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION p2025 VALUES LESS THAN (2026),
  PARTITION pmax  VALUES LESS THAN MAXVALUE
);

-- Query touching only 2024 data → partition pruning → scans p2024 only
SELECT * FROM events
WHERE  created_at BETWEEN '2024-01-01' AND '2024-12-31';

-- Drop an entire year's data instantly (no row-by-row DELETE)
ALTER TABLE events DROP PARTITION p2023;
When partitioning does NOT help
  • Queries that do not filter on the partition key — all partitions are scanned anyway (worse than a good index)
  • The table has fewer than a few million rows — plain indexes are faster and simpler
  • You need foreign keys — MySQL does not support FK on partitioned tables
  • You are using it to avoid adding a proper index — partitioning ≠ indexing
Best use case: data archiving

The killer feature of RANGE partitioning is ALTER TABLE ... DROP PARTITION. Dropping a partition is a metadata operation — it removes millions of rows instantly with no row-by-row DELETE and minimal locking. This is the primary reason to use RANGE partitioning on time-series tables.

Quick recall

RANGE by date → partition pruning on time-series queries. Best payoff: DROP PARTITION = instant archive. Limitation: no FK support; queries without partition key scan all partitions.

Connection pooling — why it matters and how to configure it

Opening a MySQL connection takes ~5–50 ms (TCP handshake + auth). At high request rates, a new connection per request collapses the DB. Connection pooling reuses connections from a managed pool.

No poolingWith pooling
Connection cost~5–50 ms per requestNear zero — connection already open
Max connectionsEasily exceeds max_connections (default 151)Bounded by pool size — predictable load
DB CPUHigh: thread creation + auth overheadLow: DB sees stable, long-lived connections
Node.js — mysql2 connection pool
// node.js / mysql2 example
const pool = mysql.createPool({
  host:            'db.internal',
  user:            'app',
  password:        process.env.DB_PASS,
  database:        'myapp',
  // Pool sizing
  connectionLimit: 10,     // max open connections at once
  waitForConnections: true,
  queueLimit:      0,     // 0 = unlimited queue (use carefully)
  // Keep connections alive
  enableKeepAlive: true,
  keepAliveInitialDelay: 0,
});
How to size the pool
  • Formula: pool_size = (core_count * 2) + effective_spindle_count (HikariCP / Percona guideline). For most cloud DBs: 10–20 per app instance is a safe starting point.
  • Too small: requests queue behind pool exhaustion → latency spikes
  • Too large: DB CPU maxes from context switching across hundreds of threads
  • With read replicas: maintain a separate pool per replica; route read queries to replicas, writes to primary
ProxySQL and PgBouncer

For larger deployments, a connection proxy (ProxySQL for MySQL) sits between the app and DB. It multiplexes thousands of app connections onto a small DB connection pool, handles failover, and can route queries (writes to primary, reads to replicas) transparently.

Quick recall

Pool = reuse open connections. Size: ~10–20 per app pod. Too large = DB thread saturation. Too small = queue starvation. ProxySQL adds a proxy layer for multi-instance deployments.

InnoDB Buffer Pool — tuning for read-heavy vs write-heavy workloads

The buffer pool is InnoDB’s main memory cache for data and index pages. Reducing disk I/O is the primary goal. Default is 128 MB — wholly inadequate for production.

  • General rule: allocate 70–80% of available RAM to the buffer pool
  • Read-heavy: larger buffer pool (keep hot data in memory); increase innodb_read_io_threads
  • Write-heavy: increase innodb_log_file_size (larger redo log); tune innodb_flush_log_at_trx_commit=2 for a durability-vs-speed tradeoff; increase innodb_write_io_threads
Key config (my.cnf) — 16 GB RAM server
innodb_buffer_pool_size      = 12G  -- ~75% of RAM
innodb_buffer_pool_instances = 8    -- improves concurrency
innodb_log_file_size         = 2G   -- larger for write-heavy
innodb_flush_log_at_trx_commit = 1  -- ACID safe (default)
-- =2 speeds writes but risks ~1 sec data loss on crash
Monitor hit rate

Run SHOW STATUS LIKE 'Innodb_buffer_pool_read%'. A healthy buffer pool hit ratio is 99%+. If it drops below that, consider increasing pool size or adding read replicas.

Quick recall

Buffer pool = keep hot pages in RAM to avoid disk reads. Sizing it correctly is one of the highest-impact MySQL tuning moves.

T22

Covering index — eliminating table lookups

A covering index contains all columns the query needs in its leaf pages — MySQL never reads the actual table row. Visible as Extra: Using index in EXPLAIN.

Covering Index Example
-- Query: order summary per user for a given status
SELECT user_id, status, COUNT(*) AS cnt
FROM orders
WHERE user_id = 42
  AND status = 'delivered'
GROUP BY user_id, status;

-- Covering index: query only needs user_id and status
CREATE INDEX idx_covering
ON orders (user_id, status);
-- EXPLAIN Extra: Using index ← no table lookup!

-- If SELECT also includes 'amount', extend the index:
CREATE INDEX idx_covering_v2
ON orders (user_id, status, amount);
Tradeoff

Wider covering indexes cost more on writes and consume more storage. Build covering indexes only for hot, frequently executed queries — not for everything.

Quick recall

Covering index = all needed data lives in the index leaf. No table row fetch needed = fastest possible read. Goal: Using index in EXPLAIN.
