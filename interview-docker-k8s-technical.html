Chapter 1 — Docker Internals
T1 Core

Containers vs VMs — Docker architecture, daemon, images, registry

A container packages app code + dependencies + config into a portable, isolated unit. Unlike VMs, containers share the host OS kernel — making them lightweight, fast to start (milliseconds vs minutes), and efficient.

text — Architecture comparison
Virtual Machines:
[ App A ]  [ App B ]  [ App C ]
[ Guest OS ][ Guest OS ][ Guest OS ]  ← each has full OS (GBs)
[ Hypervisor (VMware / VirtualBox) ]
[ Host OS ] [ Hardware ]

Containers:
[ App A ]  [ App B ]  [ App C ]
[ Libs  ]  [ Libs  ]  [ Libs  ]      ← only app + deps (MBs)
[ Docker Engine / Container Runtime ]
[ Host OS Kernel ]  ← SHARED — no guest OS!
[ Hardware ]
ComponentRole
Docker ClientCLI tool — sends commands to daemon via REST API
Docker Daemon (dockerd)Background process — manages images, containers, networks, volumes
Docker ImageRead-only blueprint — layers of filesystem changes
Docker ContainerRunning instance of an image — adds a writable layer on top
Docker RegistryStores images — Docker Hub, ECR, GCR, private registries
containerdUnderlying OCI-compliant container runtime used by dockerd
Say this in the interview

"Containers are process isolation via Linux namespaces and cgroups — not full virtualization. They start in milliseconds because there's no OS to boot. One Docker host can run hundreds of containers vs tens of VMs. In production, we don't run dockerd directly — we use containerd or CRI-O under Kubernetes."

Quick recall

VM = full guest OS per app (GBs, minutes to start). Container = shared host kernel, only app + libs (MBs, milliseconds to start). Docker daemon = manages lifecycle. Docker client = API calls only. containerd = the actual runtime under K8s. Containers use namespaces (isolation) + cgroups (resource limits).

T2 Core

Dockerfile best practices — multi-stage builds, layer optimization, production-grade images

Multi-stage builds are the single biggest improvement to Dockerfile quality. A Node.js app image drops from ~1.2 GB (with build tools) to ~80 MB (just the runtime output) when build stages are separated from the production stage.

dockerfile — Production multi-stage build
# ── Stage 1: Build ──────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app

# Copy package files FIRST — cache layer, only rebuilds if deps change
COPY package*.json ./
RUN npm ci --only=production   # ci = clean install, exact lockfile versions

COPY . .
RUN npm run build

# ── Stage 2: Production ─────────────────────────────────
FROM node:20-alpine AS production
WORKDIR /app

# Security: never run as root!
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Copy ONLY what's needed from builder — no source, no dev deps
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -q -O- http://localhost:3000/health || exit 1

CMD ["node", "dist/server.js"]
PracticeWhy it matters
Multi-stage buildsFinal image has no build tools (gcc, npm dev deps) — smaller, more secure
Copy package.json first, then codeLayer cache: if source changes but deps don't, npm install reuses cached layer
Specific tags (node:20-alpine)Reproducible builds — :latest silently breaks builds
Alpine base images5 MB vs 900 MB — minimal attack surface, fewer CVEs
Non-root USERContainer escape exploits can't get root on the host
HEALTHCHECK instructionDocker/K8s knows if app is actually healthy, not just running
.dockerignore fileExclude node_modules, .git, .env — speeds build, prevents secret leaks

Quick recall

Multi-stage = separate build from runtime, 10-15× smaller final image. Layer order = slow-changing layers first (deps), fast-changing last (source). Non-root USER = security. Specific tag = reproducibility. HEALTHCHECK = K8s probe fallback. .dockerignore = don't send secrets to build context.

T3

Docker layers & Union Filesystem — OverlayFS, Copy-on-Write, layer sharing

Every Dockerfile instruction creates a read-only layer. Docker stacks these using OverlayFS (Union Filesystem). When a container runs, a thin writable layer is added on top. All writes go there via Copy-on-Write — the underlying image layers are never modified.

text — Layer stack
[ Container Writable Layer ]   ← temporary, lost when container stops
[ COPY . . Layer           ]   ← your source code
[ RUN npm install Layer    ]   ← node_modules
[ COPY package.json Layer  ]   ← package files
[ WORKDIR /app Layer       ]
[ node:20-alpine Layer     ]   ← base OS (SHARED across all node:20-alpine images!)

Cache invalidation rule:
  Change instruction 4 → layers 4, 5, 6 ... all rebuilt
  Change only source code (last COPY) → only that layer rebuilt ✓
bash — Layer inspection and optimization
# Inspect layers and their sizes
docker history myapp:latest          # show each layer + size
docker image inspect myapp:latest    # full layer manifest

# Combine RUN commands to minimize layer count
# Bad: 3 separate layers
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get clean

# Good: single layer (cleanup in SAME layer or it doesn't shrink image!)
RUN apt-get update \
  && apt-get install -y curl \
  && apt-get clean \
  && rm -rf /var/lib/apt/lists/*
Layer sharing is the efficiency win

If 20 containers use the same node:20-alpine base, that base is stored only once on disk and RAM. Copy-on-Write means writing to a file in a container copies it to the writable layer first — reads are O(1), writes have slight overhead. In CI, push to a registry between builds so layer caches persist across runners.

Quick recall

Each Dockerfile instruction = one read-only layer. OverlayFS = stacks layers. Writable container layer = temporary, lost on stop. Copy-on-Write = writes go to writable layer, reads go through all layers. Layer sharing = identical base layers stored once across all images. Cache invalidation = any changed layer invalidates all layers below it.

Docker networking — bridge, host, overlay, DNS between containers

DriverUse caseCommunication
bridge (default)Containers on same hostContainer name as DNS on custom networks
hostMaximum performance, no isolationContainer uses host network stack directly
noneComplete network isolationNo network access at all
overlayMulti-host (Docker Swarm)Containers across hosts communicate
macvlanLegacy apps needing a MAC addressContainer appears as physical network device
bash — Network commands
# Create custom bridge network (better than default bridge)
docker network create --driver bridge mynet

# Containers on the same custom network resolve each other by name
docker run --network mynet --name api myapp
docker run --network mynet --name db postgres
# API container reaches DB via: postgres://db:5432 (name = DNS!)

# Publish ports: host:container
docker run -p 8080:3000 myapp                  # 0.0.0.0:8080 → container:3000
docker run -p 127.0.0.1:8080:3000 myapp        # bind localhost only!

docker network ls
docker network inspect mynet
Custom network vs default bridge

The default bridge network does NOT provide DNS by container name — you'd need to hardcode IPs. Always create a custom bridge network: it gives automatic DNS resolution by service name. In Docker Compose, all services get their own isolated network with name-based DNS automatically.

Quick recall

Default bridge = no name DNS. Custom bridge = name-based DNS. host mode = container shares host network (no port mapping needed, no isolation). Bind ports to 127.0.0.1 in production, not 0.0.0.0. Docker Compose auto-creates a custom network per compose file — services talk by service name.

T5

Docker volumes & storage — volumes vs bind mounts vs tmpfs

TypeStored inManaged byBest for
Volume/var/lib/docker/volumes/DockerProduction data — databases, uploads, persistent app data
Bind mountAny host path you specifyHost OSDevelopment: live code reload, config injection
tmpfsHost RAM (never written to disk)DockerSecrets in memory, temp data, high-speed scratch space
bash — Volume commands
# Named volume (preferred for production)
docker volume create pgdata
docker run -v pgdata:/var/lib/postgresql/data postgres

# Bind mount (dev: live code changes, no rebuild needed)
docker run -v $(pwd)/src:/app/src -v /app/node_modules myapp

# tmpfs — never written to disk (great for secrets)
docker run --tmpfs /run/secrets:ro,size=65536k myapp

# Volume management
docker volume ls
docker volume inspect pgdata
docker volume rm pgdata
docker volume prune         # remove unused volumes ⚠

# Backup a named volume
docker run --rm -v pgdata:/data -v $(pwd):/backup \
  alpine tar czf /backup/pgdata-backup.tar.gz /data
Volumes in production vs development

Named volumes for production — managed by Docker, good performance, portable. Bind mounts for development — mount your source code so file changes reflect immediately without rebuilding. Never use bind mounts in production: the host path creates tight coupling. tmpfs is underused and perfect for sensitive temporary data like decrypted secrets that must never touch disk.

Quick recall

Volume = Docker-managed, production-safe, easy backup. Bind mount = host path, dev only. tmpfs = RAM only, never on disk, for secrets or temp data. Backup volumes with docker run --rm + tar. docker volume prune is dangerous — removes ALL unnamed volumes.

T6 Core

Docker Compose — depends_on with healthcheck, resource limits, secrets

yaml — Production docker-compose.yml
version: '3.9'
services:
  app:
    build:
      context: .
      target: production          # multi-stage target
    ports: ["3000:3000"]
    environment:
      NODE_ENV: production
      DB_HOST: db                  # DNS name = service name
    depends_on:
      db:
        condition: service_healthy  # wait for healthcheck, not just container start!
    restart: unless-stopped
    deploy:
      resources:
        limits:        {cpus: '0.5', memory: 512M}
        reservations:  {memory: 256M}

  db:
    image: postgres:16-alpine
    volumes: [pgdata:/var/lib/postgresql/data]
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    secrets: [db_password]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  pgdata:

secrets:
  db_password:
    file: ./secrets/db_password.txt
depends_on without condition is a footgun

The common mistake is depends_on: db without a condition — the app container starts but the database process isn't ready yet, causing connection errors at startup. Use condition: service_healthy with a proper healthcheck so Docker Compose waits for the DB to actually accept connections before starting the app.

Quick recall

depends_on: condition: service_healthy = wait for healthcheck pass (not just container start). Always set resource limits — a runaway container can OOM-kill the host otherwise. Use POSTGRES_PASSWORD_FILE + Docker secrets instead of plain ENV for credentials. restart: unless-stopped for auto-recovery.

T7 Advanced

Docker security — image scanning, secrets management, capability dropping, read-only FS

bash — Security hardening commands
# 1. Scan images for CVEs (integrate into CI)
docker scout cves myapp:latest
trivy image myapp:latest
snyk container test myapp:latest

# 2. Read-only filesystem — attacker can't write malware
docker run --read-only \
  --tmpfs /tmp \          # allow writable /tmp in RAM only
  myapp

# 3. Drop ALL Linux capabilities, add only what's needed
docker run --cap-drop ALL \
           --cap-add NET_BIND_SERVICE \
           myapp

# 4. Prevent privilege escalation
docker run --security-opt no-new-privileges myapp

# 5. Limit syscalls with a seccomp profile
docker run --security-opt seccomp=/path/to/profile.json myapp

# 6. Secrets: NEVER use ENV vars — visible in docker inspect!
# Bad: exposed in docker inspect output and process lists
docker run -e DB_PASSWORD=supersecret myapp

# Good: mount as file
docker run --secret id=db_pwd,src=/path/to/secret myapp
# Available in container at /run/secrets/db_pwd
RuleImpact
Non-root user in DockerfileContainer escape can't get host root
Minimal base image (distroless / alpine)Fewer CVEs, smaller attack surface
Scan images in CI pipelineCatch vulnerabilities before production
Read-only filesystemAttacker can't write malware to container
Drop capabilitiesLimit what the container can do to the host kernel
No secrets in ENV / image layersSecrets not exposed via docker inspect

Quick recall

ENV vars = visible in docker inspect, process lists, image history. Use Docker secrets or vault mounts. --read-only = no filesystem writes (good default). --cap-drop ALL --cap-add needed = minimal kernel attack surface. Scan images with Trivy in CI. Non-root user = defense in depth against container escape.

T8

Container resource limits — CPU, memory, OOMKilled, docker stats

bash — Resource limit flags
# Memory limits
docker run --memory 512m \              # hard limit
           --memory-reservation 256m \  # soft limit (scheduling hint)
           --memory-swap 512m \         # = memory → disables swap entirely
           myapp
# If exceeds memory limit → OOMKilled (exit code 137)

# CPU limits
docker run --cpus "0.5" myapp           # max 50% of 1 CPU
docker run --cpus "2"   myapp           # max 2 CPU cores
docker run --cpu-shares 512 myapp       # relative weight (default 1024)

# Monitor live usage
docker stats                            # live CPU/mem/net/disk per container
docker stats --no-stream                # one-shot snapshot

# Debug OOMKill
docker inspect container_id --format '{{.State.OOMKilled}}'  # → true/false
docker inspect container_id --format '{{.State.ExitCode}}'   # → 137 = SIGKILL
Always set memory limits

Without memory limits, a memory leak in one container can OOM-kill the Docker host and take down ALL containers. Set --memory-swap equal to --memory to disable swap — swap causes unpredictable latency spikes. CPU limits throttle (don't kill); memory limits kill. In K8s, these map directly to resources.limits in the pod spec.

Quick recall

Memory limit exceeded → OOMKilled, exit code 137. CPU limit exceeded → throttled, not killed. --memory-swap = --memory disables swap. docker stats for live monitoring. In K8s: requests = scheduling, limits = enforcement. Always set both in production.

T9

Docker registry & image management — tagging, multi-arch, lifecycle policies

bash — Tagging, pushing, multi-arch
# Tagging strategy: never use :latest in production!
docker build -t myapp:1.2.3 .                    # semantic version
docker build -t myapp:${GIT_SHA} .               # git SHA = best for CI/CD traceability
docker build -t myapp:${GIT_SHA} -t myapp:latest . # tag both

# Push to AWS ECR
docker tag myapp:1.2.3 123456.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3
docker push 123456.dkr.ecr.us-east-1.amazonaws.com/myapp:1.2.3

# Multi-architecture build (amd64 for servers + arm64 for Graviton/M-series)
docker buildx create --use
docker buildx build --platform linux/amd64,linux/arm64 \
  -t myapp:1.2.3 --push .

# Cleanup — prevent disk filling up in CI
docker image prune -f                            # remove dangling (untagged) images
docker image prune -a --filter "until=48h"       # remove images older than 48h
docker system prune -a -f                        # nuclear: remove everything unused
Git SHA tagging in CI/CD

Tag images with the git commit SHA — it creates a direct link between the running container and the exact code version. Combined with ECR lifecycle policies (keep last 10 tagged images, delete untagged after 1 day, delete anything older than 30 days), you get free disk cost control. Multi-arch builds are now essential: AWS Graviton (arm64) gives 40% better price-performance.

Quick recall

:latest in production = reproducibility failure. Git SHA tag = exact code traceability. Semantic version tag = for versioned releases. Multi-arch buildx = one push, works on amd64 and arm64. ECR lifecycle policies = auto-delete old images. Never push secrets into image layers.

T10

Docker troubleshooting — crash debugging, exit codes, network diagnosis

bash — Debug workflow
# ── Container inspection ──
docker ps -a                                    # all containers incl. stopped (see exit codes)
docker logs container_id                        # stdout/stderr
docker logs -f --tail 100 container_id          # follow last 100 lines
docker inspect container_id                     # full config: network, mounts, env, state
docker events                                   # real-time Docker event stream

# ── Shell into a running container ──
docker exec -it container_id sh                 # alpine (no bash)
docker exec -it container_id bash               # debian/ubuntu
docker exec -it container_id sh -c "env | grep DB"  # check env vars

# ── Debug a CRASHED container (can't exec in) ──
docker run -it --entrypoint sh myapp:latest     # override entrypoint to get a shell
docker cp container_id:/app/logs ./logs         # copy files out before container dies

# ── Network debugging from inside the cluster ──
docker run --rm --network mynet nicolaka/netshoot \
  curl http://api-service:3000/health           # test connectivity from same network

# Exit code reference:
# 0   = clean exit
# 1   = application error
# 137 = OOMKilled (SIGKILL)
# 143 = graceful shutdown (SIGTERM)
# 125 = docker run itself failed
Debugging workflow

1. docker ps -a — see exit code. 2. docker logs — see the error (solves 90% of issues). 3. docker inspect — check env vars, volumes, network. 4. docker run --entrypoint sh — if container crashes on start, get a shell and run the start command manually. The nicolaka/netshoot image has curl, dig, nslookup, netstat — invaluable for network debugging.

Quick recall

Exit code 137 = OOMKilled. Exit code 143 = SIGTERM (graceful). Start debugging with logs → inspect → shell. For crashed containers: override entrypoint. For network debugging: nicolaka/netshoot on the same network. docker events for real-time lifecycle events.

Chapter 3 — Kubernetes Architecture
T11 Core

K8s architecture — control plane components, worker nodes, reconciliation loop

text — Cluster architecture
CONTROL PLANE
├── kube-apiserver      ← single entry point; all kubectl commands hit here
├── etcd                ← distributed KV store; ALL cluster state lives here
├── kube-scheduler      ← assigns pods to nodes (resource-aware, no pods run here)
├── controller-manager  ← runs all controllers: ReplicaSet, Deployment, Node...
└── cloud-controller    ← talks to cloud APIs (AWS/GCP/Azure) for LBs, volumes

WORKER NODE
├── kubelet             ← agent; ensures pod containers are running and healthy
├── kube-proxy          ← manages iptables/IPVS rules for Service routing
└── container runtime   ← containerd/CRI-O — actually pulls images and runs containers

Request flow:
  kubectl apply → API Server → etcd (store desired state)
               → Scheduler (assign pod to node)
               → kubelet (pull image, start container)
               → Controller Manager (watch actual vs desired → reconcile)
etcd is the cluster brain

etcd stores ALL cluster state — desired state, actual state, configs, Secrets. If etcd is lost without a backup, the cluster is unrecoverable. Best practice: 3 or 5 etcd nodes (odd number for Raft quorum), regular automated backups to S3, cross-AZ placement. Controllers run a continuous reconciliation loop: compare desired state (etcd) vs actual state → make them match. This is the fundamental K8s pattern.

Quick recall

apiserver = REST gateway for everything. etcd = cluster truth (back it up!). scheduler = decides which node, doesn't run pods. kubelet = per-node agent that actually runs pods. kube-proxy = iptables/IPVS for Service routing. controller-manager = reconciliation loops. Reconciliation = desired (etcd) vs actual → converge.

T12 Core

Pods, Deployments, ReplicaSets — hierarchy, lifecycle, rolling updates

Pod = smallest deployable unit; 1+ containers sharing a network namespace and storage. Pods are ephemeral — they die and are replaced, never restarted in-place. ReplicaSet = ensures N replicas always running. Deployment = manages ReplicaSets, enables rolling updates and rollback. You almost never create ReplicaSets directly.

yaml — Deployment with zero-downtime rolling update
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # max extra pods during update
      maxUnavailable: 0  # never reduce below desired count — zero downtime!
  template:
    metadata:
      labels: {app: api}
    spec:
      containers:
      - name: api
        image: myapp:abc1234       # git SHA, not :latest!
        resources:
          requests: {cpu: "100m", memory: "128Mi"}
          limits:   {cpu: "500m", memory: "512Mi"}

# Key rollout commands
kubectl rollout status deployment/api-deployment   # wait for completion
kubectl rollout undo deployment/api-deployment     # rollback to previous RS
kubectl rollout history deployment/api-deployment  # show revisions
kubectl describe pod pod-name                      # events, status, resources
maxSurge=1 maxUnavailable=0

This is the zero-downtime rolling update pattern: K8s always keeps the full desired count running, adds one new pod, waits for it to pass readiness checks, then removes one old pod. Never update a Deployment image via kubectl set image directly in production — update the YAML in Git and apply via CI/CD for full auditability. Always run kubectl rollout status in your pipeline to confirm completion.

Quick recall

Pod → ReplicaSet → Deployment hierarchy. Pod = ephemeral, replaced not restarted. ReplicaSet = N replicas always. Deployment = manages RSes + enables rolling updates. maxUnavailable=0 + maxSurge=1 = zero-downtime. Rollback = kubectl rollout undo (switches Service to old ReplicaSet). Labels = how Services find pods.

[ Pod A ] → GET http://api-service:3000 →
[ CoreDNS ] resolves → [ ClusterIP: 10.96.45.12 ]

Production pattern: ClusterIP + Ingress

Use ClusterIP for internal services and a single Ingress for external access — don't create one LoadBalancer Service per microservice (costs $20+/month each on AWS). ClusterIP + nginx-ingress Controller = one cloud load balancer, path-based routing to all services. Headless Services are essential for StatefulSets — they give stable individual DNS names (redis-0.redis-svc, redis-1.redis-svc).

Quick recall

ClusterIP = internal stable VIP. kube-proxy = iptables/IPVS rules that route to real pod IPs. CoreDNS = cluster DNS. Same namespace: service-name. Cross-namespace: service.namespace.svc.cluster.local. Headless = no VIP, DNS returns pod IPs directly (StatefulSets). Never create one LoadBalancer per microservice.

T14 Core

ConfigMaps & Secrets — consumption patterns, encryption at rest, secret rotation

yaml + bash — ConfigMap and Secret patterns
# ── ConfigMap ──
kubectl create configmap app-config \
  --from-literal=LOG_LEVEL=info \
  --from-file=config.yaml

# Consume as env var
env:
- name: LOG_LEVEL
  valueFrom:
    configMapKeyRef: {name: app-config, key: LOG_LEVEL}

# Or mount as files (updates live without pod restart!)
volumeMounts:
- {name: config-vol, mountPath: /etc/config}
volumes:
- name: config-vol
  configMap: {name: app-config}

# ── Secret (base64 encoded, NOT encrypted by default!) ──
kubectl create secret generic db-creds \
  --from-literal=password=supersecret

# Production: External Secrets Operator + AWS Secrets Manager
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
spec:
  secretStoreRef: {name: aws-secrets-manager}
  data:
  - secretKey: db-password
    remoteRef: {key: prod/myapp/db, property: password}
  refreshInterval: 1h         # auto-sync from AWS Secrets Manager
K8s Secrets are not encrypted by default

K8s Secrets are only base64 encoded — anyone with kubectl or etcd access can decode them instantly. In production: use External Secrets Operator to sync from AWS Secrets Manager or HashiCorp Vault so actual values never live in etcd. Enable encryption at rest (KMS provider in API server config). Use separate namespaces and RBAC to restrict who can get secrets.

Quick recall

ConfigMap = non-sensitive config, env var or volume mount. Volume mount = live update without pod restart. Secret = base64 only, not encrypted. External Secrets Operator = sync from Vault/AWS SM into K8s Secret. Enable encryption at rest via KMS. Secret rotation: update in secret store → ESO syncs → mounted volumes update live, ENV vars need pod restart.

PersistentVolumes & StatefulSets — stable identities, ordered lifecycle

text — PV architecture
[ StorageClass ]   ← defines provisioner (AWS EBS, GCP PD, NFS)
      ↓
[ PersistentVolumeClaim ]  ← pod requests storage (size, access mode)
      ↓ binds to
[ PersistentVolume ]  ← actual storage resource (auto-provisioned)
      ↓ backed by
[ AWS EBS / GCP PD / NFS / Ceph ]

StatefulSet pods — stable names, stable storage:
[ redis-0 ] ──→ [ PVC: redis-data-redis-0 ]  ← always this PVC
[ redis-1 ] ──→ [ PVC: redis-data-redis-1 ]  ← always this PVC
DNS: redis-0.redis-svc, redis-1.redis-svc (stable even after restart!)
StatefulSetDeployment
Pod namesStable: redis-0, redis-1Random: redis-abc12, redis-def34
DNSStable per-pod DNS via headless ServiceOne ClusterIP for all pods
StorageEach pod gets its own PVC (stable binding)All pods share the same PVC (or none)

Run databases outside K8s if possible

StatefulSets give each pod a stable identity — redis-0 always gets the same PVC and DNS name even after restart. This is essential for databases that need stable membership. However, in most companies I'd recommend managed databases (RDS, ElastiCache, DocumentDB) outside K8s — the operational overhead of stateful workloads in K8s is significant unless you have dedicated platform engineers.

Quick recall

StatefulSet = stable pod names + DNS + own PVC per pod + ordered lifecycle. Deployment = stateless, any pod replaceable. StorageClass → PVC → PV → cloud disk. RWO = single node access (EBS). RWX = multi-node (EFS/NFS). Headless Service + StatefulSet = per-pod DNS (pod-0.svc). Prefer managed DBs outside K8s.

HPA, VPA & Cluster Autoscaler — three layers of autoscaling

HPAPod count (replicas)CPU%, Memory%, custom metrics (RPS, queue depth)Traffic spikes, variable load
VPAPod CPU/memory requestsHistorical resource usageRight-size over/under-provisioned pods

yaml — HPA with custom metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef: {name: api-deployment}
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70        # scale up if avg CPU > 70%
  - type: Pods
    pods:
      metric: {name: http_requests_per_second}   # from Prometheus adapter
      target:
        type: AverageValue
        averageValue: "100"
The autoscaling chain

HPA detects high CPU → scales pods → if no node capacity, pods stay Pending → Cluster Autoscaler detects Pending pods → adds a new node → pods schedule. This full chain takes 2–5 minutes. Always maintain a warm buffer: minReplicas ≥ 2 and a spare pre-provisioned node. Don't use HPA and VPA on the same resource metric simultaneously — use VPA in Recommendation mode only.

Quick recall

HPA = pod count, reactive to load. VPA = pod resource requests, corrects sizing. Cluster Autoscaler = node count, reacts to Pending pods. Chain: HPA → pending pods → CA adds node → scheduler assigns. 2–5 min total. minReplicas ≥ 2 for HA. Custom metrics via Prometheus Adapter (KEDA is simpler for queue-based scaling).

Ingress & Ingress Controllers — TLS, path routing, cert-manager, rate limiting

yaml — Ingress with auto-TLS and path routing
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rate-limit: "100"           # 100 req/s per IP
    nginx.ingress.kubernetes.io/use-regex: "true"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"       # auto TLS certificate!
spec:
  ingressClassName: nginx
  tls:
  - hosts: [api.myapp.com]
    secretName: api-tls-cert           # cert-manager auto-populates this
  rules:
  - host: api.myapp.com
    http:
      paths:
      - path: /api/v1
        pathType: Prefix
        backend:
          service: {name: api-service, port: {number: 3000}}
      - path: /static
        pathType: Prefix
        backend:
          service: {name: static-service, port: {number: 80}}
Ingress vs Service LoadBalancer

The Ingress object is just routing rules — the Ingress Controller (nginx, traefik, AWS ALB Controller) is the actual implementation. One Ingress Controller handles all services, so you pay for one cloud load balancer instead of one per service. cert-manager + Let's Encrypt = zero manual TLS certificate management, auto-renewal. nginx annotations let you add rate limiting, auth, CORS, and rewrite rules directly on the Ingress.

Quick recall

Ingress = routing rules (not the implementation). Ingress Controller = nginx/traefik/ALB that implements them. One controller = one LB for all services. cert-manager + Let's Encrypt = auto TLS issuance and renewal. Path-based routing splits traffic to different Services. Annotations configure nginx behavior per-Ingress.

Deployment strategies — Rolling Update, Blue/Green, Canary with Argo Rollouts

text — Strategy comparison
Rolling Update (default):
  [v1][v1][v1] → [v2][v1][v1] → [v2][v2][v1] → [v2][v2][v2]
  ✓ Zero downtime  ✓ Gradual  ✗ Brief mix of v1+v2 in prod

Blue/Green:
  Blue(v1):  [v1][v1][v1]  ← live traffic
  Green(v2): [v2][v2][v2]  ← deploy, test, then flip Service selector
  ✓ Instant rollback (flip selector back)  ✗ Double resource cost

Canary:
  v1 pods: 9  (90% traffic)
  v2 pods: 1  (10% traffic — canary)
  Monitor → increase v2 weight → eventual 100%
  ✓ Minimal blast radius  ✓ Real-user validation  ✗ Requires traffic splitting
yaml — Canary with Argo Rollouts
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 10          # 10% to new version
      - pause: {duration: 5m}  # watch metrics for 5 minutes
      - setWeight: 50          # 50% to new version
      - pause: {duration: 10m}
      - setWeight: 100         # full rollout
      analysis:
        templates:
        - templateName: error-rate   # auto-rollback if error rate spikes!
Choosing a strategy

Rolling Update = default, sufficient for most changes. Blue/Green = major breaking changes where instant rollback is critical (just flip the Service selector). Canary = highest-risk changes: send real user traffic to the new version at low percentage, automated Prometheus analysis decides whether to proceed or roll back. Argo Rollouts handles the traffic split and analysis automatically.

Quick recall

Rolling = default, gradual, v1+v2 briefly coexist. Blue/Green = full parallel deployment, instant rollback via selector flip, 2× resources. Canary = low % to new version, automated analysis, rollback on metric breach. Argo Rollouts = setWeight + pause + AnalysisTemplate = production canary. All three give zero downtime.

Probes & resource QoS — liveness, readiness, startup, Guaranteed vs Burstable

yaml — All three probes + resources
containers:
- name: api
  image: myapp:latest
  resources:
    requests:               # used by scheduler to find a fitting node
      cpu: "100m"            # 0.1 CPU cores
      memory: "128Mi"
    limits:                 # hard cap enforced by cgroups
      cpu: "500m"            # throttled if exceeded (not killed)
      memory: "512Mi"       # OOMKilled if exceeded

  startupProbe:             # delays liveness/readiness for slow-starting apps
    httpGet: {path: /health, port: 3000}
    failureThreshold: 30
    periodSeconds: 10        # 5-minute window for app to start

  livenessProbe:            # is app alive? restart container if not
    httpGet: {path: /health, port: 3000}
    initialDelaySeconds: 10
    periodSeconds: 15
    failureThreshold: 3

  readinessProbe:           # can app take traffic? remove from LB endpoints if not
    httpGet: {path: /ready, port: 3000}
    periodSeconds: 5
    failureThreshold: 2
ProbeFailure actionUse for
LivenessRestart the containerDetect deadlocks, infinite loops, unrecoverable state
ReadinessRemove pod from Service endpoints (stays running)DB connection not ready, cache warming, slow request in flight
StartupDelays liveness/readiness checksSlow-starting apps: JVM, ML model loading, large migrations
QoS classConditionEviction priority
Guaranteedrequests == limits for all resourcesLast evicted under node pressure
Burstablerequests < limitsMiddle priority
BestEffortNo requests or limits setFirst evicted under memory pressure
Liveness vs Readiness — don't mix them up

Liveness restarts a broken pod (deadlock recovery). Readiness removes a temporarily busy pod from load balancing — it stays running. Never use the same endpoint for both: readiness should check if the app can handle requests right now (DB connected, cache warm); liveness should just check if the process is alive. Use separate /health and /ready endpoints.

Quick recall

Liveness = restart. Readiness = remove from LB. Startup = delay the others for slow apps. requests = scheduling. limits = enforcement. CPU limit = throttle. Memory limit = OOMKill. QoS: Guaranteed (req=lim) → Burstable (req<lim) → BestEffort (none). Eviction order = BestEffort first, Guaranteed last. Set requests + limits on all production containers.
